## 今回参照したAI組織トレンド

### 1. Orchestrator-Subagent パターン（Anthropic社内実装）

Anthropicが自社の研究システムに採用したアーキテクチャ。リードエージェント（Orchestrator）が戦略を立て、並列で動くサブエージェントに独立したサブタスクを割り振る。Opus 4をリード、Sonnet 4をサブに使った構成で、単一エージェント比90%以上の性能向上を記録。**重要な知見**: サブエージェントへの指示が短すぎると「重複作業・誤解釈」が発生する。サブエージェントには「目的・出力フォーマット・使うツール・タスク境界」の4要素を必ず渡すこと。

### 2. Solo Founder + AI Team モデル（2025-2026 主流化）

2025年中頃、新規スタートアップの36.3%がソロファウンダー——AIコーディングエージェントの普及と完全に連動した数字。単一ファウンダーが「コーディングエージェント・マーケティングエージェント・サポートエージェント・分析エージェント」を組み合わせ、10人チーム相当のアウトプットを出す事例が急増。**コスト感**: 従来チームの2〜5%のコストで80〜85%の実行タスクをカバー。

### 3. 「コンテキストエンジニアリング」が最重要スキルに

2026年時点でのAI開発における最大の差別化要因は「プロンプトの書き方」ではなく「エージェントに何をどの順番で読ませるか」のコンテキスト設計。CLAUDE.mdによる部署別ルール定義は正しい方向だが、**エージェントが本当に必要な情報にたどり着くまでの「コンテキスト経路設計」**が次の課題。

### 4. マルチエージェントの2大アンチパターン

- **過剰分業**: 部署を増やすほど「引き継ぎコスト」が増え、スループットが下がる。単純なタスクはシングルエージェントで完結させるべき
- **状態共有の欠如**: 各エージェントがバラバラの前提で動くと、矛盾した決定が積み重なる。共有状態（=記憶庫）が唯一の解決策

### 5. 「Lean AI Pipeline」—— 1サイクルの最短化

フレームワーク選定より**1サイクル（アイデア→本番URL）の所要時間を最短化すること**が競争力の源泉。ボトルネックを特定し、そこだけに自動化を集中投下する戦略が2026年のSOTA（最良実践）。

## 再設計提言

### 部署構造（追加・統合・廃止の提言と理由）

**現状の6部署体制は正しい。ただし役割の「境界」を再定義する。**

#### 提言1: 「企画部」と「記憶庫」を接続するエージェント経路を強化

現在、企画部はWebトレンド調査と過去の記憶庫参照を別々に行っているが、**「前日のプロジェクトの何がユーザーに刺さったか」というフィードバックデータが記憶庫に入っていない**。
広報部の反響データ（いいね数・RT数・分析仮説）を `記憶庫/feedback-loop.md` として独立させ、企画部が毎晩必ず読む経路を作る。

#### 提言2: 「QA部」と「デザイン部」の責任分担を逆転させる

現在の流れ: `実装 → genius-audit → QA → デプロイ → デザイン3ループ → 再デプロイ → 最終QA`

**問題**: QAが2回発生している（デプロイ前コードQAと、デザイン改善後の最終確認）。これは引き継ぎコストが高い。

**提言**: デザイン部をデプロイ**前**に組み込む。

```
実装 → genius-audit → デザイン部レビュー（コード読まずUI設計のみ） → QA → デプロイ → デザイン3ループ
```

デザイン部の「First Impression」はコードを読まなくてもUIの設計図（仕様書の画面構成）レベルで事前に指摘できる。これにより「デプロイ後に大幅UI変更が入ってQAが再実施」という無駄を削減できる。

#### 提言3: 「広報部」を「成長ループ部」に改名・役割拡大

現在の広報部は「note記事ドラフト生成 + Xツイート案提示」に留まっている。しかし #100Day Challenge の競争力は**「毎日作って毎日外に出す」というコミットメント自体**にある。

成長ループ部の役割に以下を追加:
- **コミュニティ反響の定量分析**: いいね/RT/インプレッションを蓄積し、「どのジャンル・どの技術・どのUI」が反響を呼ぶかを統計化する
- **コンテキスト蓄積**: 分析結果を `記憶庫/audience-insights.md` に書き込み、企画部が次の企画に使えるようにする

#### 提言4: 「技術顧問」ロールを準部署として追加（軽量）

専用ディレクトリ不要。CLAUDE.mdへの記述のみで構成。役割: **「今使っているスタックが3ヶ月後も最善か」を定期的に問い直す**。
- 月1回: 使用スタック（Firebase・Vercel・Vanilla JS）の代替技術をWebSearchでリサーチ
- 新技術が優位な場合にのみ移行提言を出す
- `記憶庫/tech-decisions.md` に記録

### ワークフロー（フロー改善の具体案）

#### 問題: 積みプロジェクトが増殖する構造的原因

現在のフローに「前日のプロジェクトが未完走でも翌日新しいプロジェクトを始められる」というルールがない。

**提言: 「完走優先ルール」を本社CLAUDE.mdに明記する**

```
新規プロジェクト着手前チェック:
- 直近のプロジェクトがDoD未達の場合、新規着手より完走を優先する
- 例外: 技術的なブロッカーが発生して完走できない場合のみ、翌日プロジェクトに移行してよい
  （その場合は「ブロッカー内容」を記録すること）
```

#### 問題: デザイン3ループの「義務感」がスピードを下げている

現在のルールでは3ループ全部を毎回やることが必須になっている。しかし**ループの質は時間の長さではなく改善の有無**で測られるべき。

**提言: 3ループを「判定式」に変える**

```
Loop 1 (First Impression): 3秒で分からなければ改善。分かれば次へ。
Loop 2 (Core Flow): 3ステップ以内で始められなければ改善。始められれば次へ。
Loop 3 (Shareability): 友達に説明なしで渡せなければ改善。渡せれば完了。
```

チェックだけで問題がなければそのループは「確認済み・改善不要」として記録し、次に進む。無改善ループを「サボり」と定義しない。

#### 問題: 部署間の受け渡しタイミングが曖昧

**提言: 各部署に「開始トリガー」を明記する**

| 部署 | 開始トリガー（明文化すべき条件） |
|------|---------------------------|
| 開発部 | 企画部の仕様書が `specs/` に存在する |
| genius-audit | 開発部が「実装完了」を宣言した |
| QA部 | genius-auditがGOを出した |
| 運用部 | QA部がGOを出した |
| デザイン部 | 運用部がデプロイURLを確定した |
| 広報部 | デザイン3ループ完走 + 最終URL確定 |
| 企画部（翌朝） | 広報部の反響データが `reports/` に保存されている |

### 自動化拡張（次に自動化すべき工程）

優先順位順に3つ挙げる。

#### 自動化候補1（優先度: 高）— スケジュールタスクの動作確認と監視

現在4つのスケジュールタスクが `lastFiredAt: null` のまま。**一度も発火していない自動化は存在しないのと同じ**。

実装: 毎朝「おはよう」フレーズ処理の中に「スケジュールタスクが昨夜発火したかチェック」を追加する。発火ログがなければ即座に手動トリガーで補完する。

#### 自動化候補2（優先度: 高）— Firebaseプロジェクト初期化の半自動化

毎回1〜2時間かかる最大ボトルネック。Authコンソール手動設定だけは自動化できないが、以下は自動化できる:

```bash
# _templates/new-project.sh として作成
# 使い方: ./new-project.sh food-tracker "Food Tracker" "food-tracker-abc123"
PROJECT_NAME=$1
DISPLAY_NAME=$2
FIREBASE_ID=$3

mkdir -p "開発部/$PROJECT_NAME"
cp -r _templates/vanilla-js-starter/* "開発部/$PROJECT_NAME/"
# .firebaserc の project-id を置換
sed -i '' "s/PROJECT_ID_PLACEHOLDER/$FIREBASE_ID/" "開発部/$PROJECT_NAME/.firebaserc"
sed -i '' "s/APP_NAME_PLACEHOLDER/$DISPLAY_NAME/" "開発部/$PROJECT_NAME/manifest.json"
echo "✅ 開発部/$PROJECT_NAME を初期化しました。Firebase ConsoleでAuth設定を忘れずに。"
```

#### 自動化候補3（優先度: 中）— DoD進捗の可視化

現在「どのプロジェクトがDoDのどこまで達しているか」を把握するには人間がファイルを読んで確認する必要がある。

実装: 毎晩のスケジュールタスクに「DoD進捗マトリクス自動更新」を追加。`運用部/reports/dod-status.md` を毎晩上書きし、翌朝の「おはよう」時に表示する。

### スケール対策（100プロジェクト時代への備え）

100プロジェクト到達時（約94日後）に直面する課題と今から打てる手。

#### 課題1: 記憶庫のサイズ爆発

100プロジェクト分のパターン・決定・教訓が蓄積されると、`reusable-patterns.md` が1万行を超える可能性がある。Claude Codeのコンテキストウィンドウで全文読むことが難しくなる。

**対策**: Day 30時点で記憶庫をカテゴリ別に分割する。
```
記憶庫/
  firebase-patterns.md     # Firebase関連パターン
  ui-patterns.md           # UIコンポーネントパターン
  api-security.md          # APIキー・認証のベストプラクティス
  audience-insights.md     # X/noteの反響データ分析
  failed-experiments.md    # うまくいかなかった実験の記録
```

#### 課題2: プロジェクト検索の困難化

100プロジェクトあると「あの時作ったやつどれだっけ」が頻発する。

**対策**: Day 30時点で `企画部/project-index.md` を作成。プロジェクト名・URL・技術スタック・コアバリューをタグ付きで管理し、「Firebase + Auth を使ったやつを探したい」などの検索に答えられるようにする。

#### 課題3: デザイン3ループの惰性化

100回やると改善パターンが固定化し、「ただこなす」状態になる可能性がある。

**対策**: Day 30時点で「デザイン部の改善施策ライブラリ」を作る。過去に効果があった改善（例: 「スプラッシュスクリーン削除で初回体験改善」「CTAボタンを画面下固定に変更」）を蓄積し、新プロジェクトへの流用を加速する。

## 外部コンサルタントからの総評

この組織の最大の強みは「**設計の正確さ**」にある。6日間でフルサイクルのワークフローを言語化し、6つの部署に役割を割り振り、genius-auditゲート・Stopフック・Writeフックを稼働させた。同条件の人間チームが6日でこの水準を達成するのは難しい。

一方で最大の弱みは「**完走への意志**」の欠如ではなく、**完走を妨げる構造的バイアス**にある。新しいものを作り始める行動は即時報酬を生む。既存のものを完走させる行動は遅延報酬しか生まない。AIファーストの組織でもこのバイアスは同じように働く——むしろ「実装が速い」ぶんだけ積みが増えるスピードも速い。

上記の3つの即時改善のうち「完走優先ルール」だけが根本的な構造改善であり、他の2つは効率化だ。構造改善なしに効率化しても積みは増え続ける。

**最初に手を打つべきは「完走優先ルール」の本社CLAUDE.mdへの追記**、1文。これだけで組織の方向が変わる。
