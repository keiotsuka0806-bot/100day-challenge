### Taiichi Ohno の視点：棄却ログなき企画は知識のムダ

**思考原理**: 大野耐一のTPS（トヨタ生産方式）は「7つのムダ」を排除する。そのうち最も見えにくいのが「知識のムダ（Intellectual Muda）」——問題解決の過程で発生した学習が記録されず消えていくこと。

**現状の問題**: 企画部は毎日10件のスペックを生成し、1件が採択される（推定）。残り9件は棄却されるが、「なぜ棄却したか」が一切記録されない。この知識が蒸発するサイクルが毎日繰り返されている。100日後、なぜ「会議タイマー」より「気分予報」を選んだのか、なぜ特定のアイデアを捨てたのかの記憶がゼロになる。これは100日分の「プロデューサーとしての判断力」が揮発することを意味する。

また、IDEA_LOG.mdの「使い方」説明には「気に入ったアイデアを1つ選んで」とあるが、選んだ記録を付けるフィールドが機能していない（テーブルが空のまま）。

**推奨アクション**: `企画部/rejection-log.md` を作成し、採択しなかったアイデアの棄却理由を1行で記録する。

### W. Edwards Deming の視点：PDCAサイクルが「P→D」で止まっている

**思考原理**: Demingの「Plan-Do-Check-Act」サイクルは、Checkなしには改善にならない。多くの組織がPDCAと言いながら実際にはPD（計画と実行）だけで終わる——検証も対策もない。

**現状の問題**: Day4の改善レポートに8つの推奨アクションが列挙された。Day5の本レポート作成時点で実行確認できたのは記憶庫作成の1件のみ（14%）。最も優先度の高かった`.claude/settings.json`の作成すらゼロ。しかし誰もそれを「確認」しなかった。なぜか？——毎日の改善レポートが「Check」フェーズなしに新しい推奨を生成するからだ。今日のレポートも同じパターンを繰り返せば、明日のDay6レポートには「昨日のDay5の推奨が未実行」というエントリが追加されるだけになる。これはDeming的に言えば「PDPDPDループ」（Action/Checkが欠如）。

**推奨アクション**: このレポートテンプレートに「前回レポート実行状況チェック」セクションを冒頭に追加する（本レポートで既に実装）。さらに、`.claude/settings.json`の作成を今日中の最優先タスクとし、実行確認のBashコマンドを明記する。

**具体的な実装**:

`.claude/settings.json` の作成（今日中・最優先・所要5分）:
```bash
mkdir -p /home/user/100day-challenge/.claude
```

ファイル内容:
```json
{
  "permissions": {
    "allow": [
      "Bash(git add*)",
      "Bash(git commit*)",
      "Bash(git push*)",
      "Bash(git status*)",
      "Bash(git log*)",
      "Bash(ls*)",
      "Bash(mkdir*)",
      "Bash(date*)",
      "Bash(find*)",
      "Bash(firebase serve*)",
      "Bash(firebase deploy*)"
    ]
  }
}
```

これにより、毎日の自動エージェントがgit操作・ファイル確認・Firebaseデプロイを人間の承認待ちなしに実行できるようになる。特に本レポートのような定期実行タスクで、毎回`git commit`の権限確認ダイアログが出る問題が解消される。

運用ルール追加: 各デイリーレポートの冒頭に「前回レポート実行状況」セクションを設け、完了・未完了・延期を必ず記録する。完了率が50%未満の場合は新規推奨を追加しない（負債を積まない）。
