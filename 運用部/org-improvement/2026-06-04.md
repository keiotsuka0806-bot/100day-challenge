### Paul Graham の視点：Default Alive か Default Dead か

**思考原理**: スタートアップの生存判断は「今の軌跡を続けると生き残るか（Default Alive）、それとも死ぬか（Default Dead）」。Y Combinator で最初に聞く質問。100日スタジオも同じ問いに晒されている。

**現状の問題**: Day3時点の状態を100日に線形外挿する。記憶庫がなければ毎プロジェクトでコンテキスト設定に30分消費→100日×30分=50時間のロス。`.claude/settings.json`がなければ自動化は1件も動かない。mood-forecastのようなdeployment configの欠如が100件あれば100件分の「完成しているが使えないプロジェクト」が積み上がる。このペースは **Default Dead**——今日改善しなければ、Day30頃に制作リズムが崩壊する。逆に、今日 `.claude/settings.json` と記憶庫を作れば、以降の全セッションが高速化する。複利の効果が最大になるのは Day3 の今。

**推奨アクション**: 「今日の30分の投資が残り97日を救う」順に実行する。最初にやることは記憶庫の初期化ではなく `.claude/settings.json` の作成——これがないと他の自動化が全て手動のまま。

**具体的な実装**:

```bash
mkdir -p /path/to/repo/.claude
```

`.claude/settings.json` の初期内容:
```json
{
  "permissions": {
    "allow": [
      "Bash(git:*)",
```

これで毎回の権限確認ダイアログが削減され、自動化エージェントが人間の承認待ちでブロックされなくなる。

### Kent Beck の視点：テストを先に書け、デプロイテストも同じ

**思考原理**: TDDの本質は「動く仕様を先に定義することで、実装が何を達成すべきかを明確にする」。テストが緑になった瞬間が「完成」。この原則をプロジェクト管理に適用すると、「デプロイテスト」を最初に書くことになる。

**現状の問題**: mood-forecastの `scripts/build.js` に埋め込まれたコメント——「Workaround: Vite/Rollup can't handle '#' in paths (treated as URL fragment)」——は、**ローカル環境の暗黙の前提**（開発者のフォルダ名が `#100Day Challenge` であること）が本番環境・CI環境・他の開発者の環境に伝達されていないことを示す。この種の「環境固有のワークアラウンド」は、スタジオが将来GitHub Actionsで自動ビルドしようとした際に無音で失敗する。

**推奨アクション**: `記憶庫/lessons.md` の最初のエントリとして、このワークアラウンドを記録する。さらに、`開発部/mood-forecast/CLAUDE.md`（未作成）を新規作成し、ビルド上の注意事項を明記する。

**具体的な実装**:

`記憶庫/lessons.md` 初期内容:
```markdown
# 教訓集

## ビルド環境
- **[mood-forecast] Vite/Rollup はパスに`#`を含むと URLフラグメントと誤認識してビルド失敗する**
  - 原因: ローカル開発フォルダが `#100Day Challenge/` という名称
  - 回避策: `scripts/build.js` が `/tmp` にコピーしてからビルド
  - 教訓: 新プロジェクト作成時、`npm run build` を最初のコマンドとして実行して環境を検証すること

## Firebase Auth
- Google Sign-In popup は mobile Safari で失敗することがある → redirectベースも要検討
- onAuthStateChanged が発火する前に Firestore を呼ぶと permission-denied が出る

## Firestore
- base64 data URL で写真を保存すると document size が 1MB に近づく → Storage への移行を早めに検討
- onSnapshot リスナーは auth 状態変化で必ず detach すること（メモリリーク防止）

## vanilla JS SPA
- cloneNode() で DOM 要素のリスナーを剥がしてから再レンダリングするパターンが有効
```
