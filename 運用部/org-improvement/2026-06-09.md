### W. Edwards Deming の視点：学習ループの断絶

**思考原理**: 品質は個人の努力ではなくシステムから生まれる。改善には Plan-Do-Check-Act の閉ループが必須で、「Check」がなければどれだけ「Do」しても組織は成長しない。

**現状の問題**: `記憶庫/` が存在しないことで、Act（学びを次回に活かす）フェーズが物理的に不可能になっている。広報部 CLAUDE.md が `記憶庫/decisions.md` への書き込みを定義しているが書き込み先がない。Genius Council でせっかく知見を生成しても、`デザイン部/reports/` に孤立したままで次の企画に接続されない。「学びを蓄積する」が組織の最優先事項（Step 0 優先順位1位）なのに、その器が存在しない。

**推奨アクション**: 今日、`記憶庫/` の3ファイルをシード状態で作成し、既存の知見を10件だけ手動で移植する。

### Taiichi Ohno の視点：仕掛品（WIP）の積み上がり

**思考原理**: トヨタ生産方式の核心は「流れを止めない」こと。仕掛品（WIP）は可視化してキャップを設け、完成品が出るまで新規着手しない「プル」方式に切り替える。完成していない仕事の山は、価値を生まない待機在庫（muda）である。

**現状の問題**: 10プロジェクト中5件が「要修正」という状態は、フォードの組み立てラインに例えると「半分の車が修理待ちで工場の床を塞いでいる」状態と同じ。毎日新規プロジェクトを立ち上げながら修正バックログが解消されない構造では、要修正件数は単調増加する。現に2026-06-07時点で5件の要修正が報告されており、前日の状態は不明（比較ファイルがない）。

**推奨アクション**: 本社 CLAUDE.md の「全社共通ルール」に **WIPキャップ** ルールを1行追加する。

**具体的な実装**:

`CLAUDE.md` の「全社共通ルール」セクションに追加:

```markdown
- **WIPキャップ**: 要修正プロジェクトが3件を超えたら、新規プロジェクト立ち上げを止めて要修正の解消を優先する
```

さらに `開発部/CLAUDE.md` の「朝のフレーズ」処理に条件分岐を追加:

```
入力: 「今日は[name]を作りたい」
最初に確認: node 運用部/scripts/organization-health-check.sh
要修正が3件以上なら: 「WIPキャップ超過（現在N件）。先に修正を進めることを推奨します」と表示
```

### Paul Graham の視点：ドキュメントが実装に先行する危険

**思考原理**: 「Do things that don't scale.」スケールしないことを手動でやることで、本当に必要な自動化が何かが分かる。まだ存在しないシステムのドキュメントを詳細に書くことは、実際の問題を見えにくくする。

**現状の問題**: 運用部 CLAUDE.md には `post-deploy-genius-cycle.mjs`, `sync-project-registry.mjs`, `release-check.mjs`, `automation-push-main.sh` 等のスクリプトが詳細に記述されているが、実在するのは `organization-health-check.sh` のみ。これらのスクリプトが「存在する前提」で手順が書かれているため、新しいセッションでClaude Codeがそれらのスクリプトを実行しようとして失敗する。ドキュメントが実態より10本先を走っており、「地図と実際の道路が違う」状態。

**推奨アクション**: 運用部 CLAUDE.md の未実装スクリプトに `[未実装]` タグを付ける。同時に各スクリプトの「手動代替手順」を3ステップ以内で追記する。

**具体的な実装**:

`運用部/CLAUDE.md` の該当箇所を以下のように変更:

```markdown
### Release Check [未実装 — 手動代替あり]

スクリプトが使えない場合の手動代替:
1. 本番URLにブラウザでアクセスしHTTP 200を目視確認
2. ログイン → CRUD操作を1往復確認
3. GC-L1/L2/L3の各チケットファイルが存在し「Done」または「Skipped」であることを確認

（スクリプトが実装されたら上記の手動手順と置き換える）
```

この変更により、スクリプト未存在エラーで詰まることなく、手動で品質ゲートを通過できる。

## 業界トレンドとの照合

**2024-2025年 AI開発組織のベストプラクティスとの比較**

| トレンド | 業界標準 | 当スタジオの現状 | ギャップ |
|---------|---------|----------------|---------|
| **Agent Memory** | Mem0, LangGraph の永続化レイヤー | 記憶庫ディレクトリが未作成 | 大 |
| **LLM-in-the-loop QA** | 自動テスト + AI コードレビュー | /code-review スキルあり、実行率が低い | 中 |
| **WIP可視化** | Linear の Cycle / GitHub Projects のカンバン | organization-health.md のみ | 中 |
| **Context Window Management** | CLAUDE.md の適切な分割・参照 | 本社 CLAUDE.md に全知識が集中（WhiskyNoteアーキテクチャまで記載） | 中 |
| **Prompt-as-Code** | プロンプトをGit管理し差分レビュー | CLAUDE.md はGit管理済み ✓ | なし |
| **One-click deploy** | Vercel/Firebase の自動デプロイ | firebase deploy コマンドで十分 ✓ | なし |
| **Parallel agent tasks** | Devin, Claude Code の並行実行 | 単一セッション実行。並行化未活用 | 小〜中 |

**特に注目すべき2025年トレンド**: Anthropic の Claude Code が「Agentic Loop」でプロジェクト全体を自律実行するパターンが主流化しつつある。現在のスタジオは「Claude Code + 人間の確認」というハイブリッドだが、`記憶庫/` が充実すれば、Claude Code が過去の失敗を参照して同じミスを繰り返さない「学習する組織」に近づける。Memory-augmented agent（Mem0 パターン）の要素を、ローカルの Markdown ファイルで実現している構造は、依存なしで動くシンプルな代替として評価できる — ただし書き込まれないと意味がない。
