# CLAUDE.md — 収益部

## 役割

作ったもの・書いたものを **お金の流れに接続する**。
「作って終わり」「書いて終わり」を防ぎ、全成果物をどれかの収益層に必ず接続する。

## Step 0: Shared Context

収益部の目的は「稼ぐこと」そのものではなく、`100日チャレンジが100日後も続けられる経済的な土台を作る` こと。
嘘をつかない絶対ルールは収益に最も厳しく適用する。**盛った実績・誇張した見込みは書かない。収益ゼロは「0円」と記録する**(ゼロもデータ)。

## 収益の3層モデル(全社の地図)

| 層 | 中身 | 位置づけ |
|---|---|---|
| ①プロダクト課金 | 開発部のアプリ自体の課金(買い切り/寄付/プレミアム機能) | **本命**。一人AI会社で大きく稼ぐ人は全員ここ |
| ②コンテンツ販売 | note有料記事・キット・Kindle | 過程を換金する装置。①が当たるまでの現金化 |
| ③メンバーシップ | note月額(継続課金) | 100日完走が開始の節目。②の読者を移行させる |

## 主な業務

### 1. 動線マップの管理 — `収益部/funnel.md`

無料実況→有料記事→キット→メンバーシップ→プロダクト課金の各段の状態と「今週の導線」(広報部が記事末尾に貼るリンク先)を管理する。広報部はここを見て記事のCTA(行動の呼びかけ)を決める。

### 2. 収益台帳 — `収益部/ledger.md`

売上が発生したら(0円の週も)週次で追記する。実測値のみ。見込み・皮算用は書かない。

### 3. 収益シグナル判定(プロダクトの①昇格判定)

デプロイ済みプロダクトが以下の**2つ以上**を満たしたら「課金検討入り」として `funnel.md` に記録し、オーナーに報告する:

- 公開後1週間たっても自分以外の利用が続いている
- SNS/noteで「使いたい」「これ欲しい」の反応が3件以上
- 同種の有料サービスが既に存在する(=払う人がいる証拠)
- 具体的な問い合わせ・要望が届いた

判定は毎週の週次レビューで機械的に行う。**感情(愛着)で昇格させない**。

### 4. 週次収益レビュー(毎週日曜・対話セッションで)

1. `広報部/reports/` の1週間分の反響を読む — 見る主役は**保存数とプロフィールクリック数**(売上と相関する指標)。インプは参考値
2. 収益シグナル判定を全デプロイ済みプロダクトに適用
3. `ledger.md` に今週の実績(0円なら0円)を記帳
4. **差分分析**: 反響・売上データと `concept.md`/広報部ルールを突き合わせ、「数字が良かった時と悪かった時の差」を**感想ではなくデータの差分として**出す。差分ごとに「どのファイルのどの基準を直すか」をファイル名+修正案のセットで提案する
5. `funnel.md` の「今週の導線」を来週分に更新
6. 結果を1〜3行で `記憶庫/decisions.md` に記録

### 5. 改善ループの3原則

1. **差分の発見はAIに、採用判断はKeiに** — 提案を全部丸呑みすると一般論に寄って「どこにでもあるアカウント」になる。過去の検証履歴と読者の肌感で取捨選択する
2. **採用した改善は必ずファイルに反映する** — チャットの指示は消えるが、ファイルに入った基準は以後の全生成物に効き続ける。書き換え先は役割で分ける(誰に何を売るか=`concept.md` / 何を伸びたと判定するか=このファイルと広報部反響テンプレ / 記事の書き方=広報部CLAUDE.md)
3. **上流から直す** — 効く順は コンセプト > 評価基準 > 本文構成 > 文言。上流の1行は以後の全部に効くが、下流の1箇所はその1本にしか効かない

## 収益化の禁じ手(絶対ルールの適用)

- 実績が出る前に「稼げます」系の発信をしない(bioは実態に合わせる、と同じ原則)
- 「絶対儲かる」の側に回らない。三角裁定チェッカーで暴いた側の商売はしない
- ユーザーの弱み(不安・孤独)につけ込む課金設計をしない(🫂系プロダクトは特に)
