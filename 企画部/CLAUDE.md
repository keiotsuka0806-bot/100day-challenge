# CLAUDE.md — 企画部

## 役割

アイデアを「作れる仕様」に変換する。実装を始める前に必ずここを通す。

## Step 0: Shared Context

企画部の役割は、ただ案を増やすことではない。`100日チャレンジを通してAI組織そのものを育てる` ために、学びが残る企画を作ること。

優先順位は次の順に扱う。

1. 既存の学び、反響、失敗を次の企画に接続する
2. 1日あたり1プロジェクトの共有可能な成果に変わる企画を出す
3. 企画の混入や重複を防ぐ

企画を作る前に、必要なら `広報部/CLAUDE.md` の反響データ、`運用部/reports/` の運用知見、`記憶庫/` の学びを先に読む。

（企画書は前夜に作成済み）

**重要：必ず10案すべてを表示すること。5案や8案で終わらない。**


## 夜のフレーズ — 毎晩0時に自動実行（スケジュール済み）

毎晩深夜0時に自動実行。**明日の10案企画を生成する**。

```
トリガー: 毎晩0時（自動・スケジュール済み）
処理: 明日の日付で10案企画を作成
出力: specs/YYYY-MM-DD-idea1.md
      specs/YYYY-MM-DD-idea2.md
      ...
      specs/YYYY-MM-DD-idea10.md （計10ファイル）
```

**ユーザーが言う必要なし** — CronCreate で設定済み

### 10案の選定基準
- **難度の幅** — 低〜高を混ぜる（低の数は固定しない。「低5案必須」は2026-06に廃止済み）
- **多様性** — Web、モバイルアプリ、ツール、ゲームなど様々なジャンル
- **面白さ** — 2-3件は「ユーザーに響きそう」「並行改善の価値あり」と思う企画を混ぜる
- **技術バリエーション** — Vanilla JS、React、Firebase など複数の技術スタックを試す
- **収益の芽** — 各案に、下の「収益の型」表のどの型(A〜D)に当てはまり成立条件を満たすかを一言添える(層は `収益部/CLAUDE.md` の3層モデル参照)。どの型にも当てはまらなければ「なし」と正直に書く。「なし」が10案続いたら企画の方向を収益部と見直す

### 収益の型と成立条件（2026-07-09 実例ディープリサーチ反映）

「収益の芽」は願望でなく、実例で成立が確認された型に当てはめて書く。詳細と出典は `企画部/リサーチ/2026-07-09-一人AI会社収益化リサーチ.md`。

| 型 | 成立条件（実例ベース） | 検証済み実例 |
|---|---|---|
| A. 課金型（サブスク/買い切り） | **「あったらいい」でなく「ないとこまる」**課題解決。払う人の職業・状況が具体的に言える | KOSAKAI氏・教員向け学習ツール（2025年度利益391万円） |
| B. 広告型（無料toC） | 大量トラフィック前提。目安=1日1万UUで月10万円級。課金転換は0.1%程度しか期待できない | Uzu氏・Pomofocus（月100万UU、収益の8割が広告） |
| C. キット/テンプレ販売 | 「自分が通った道」を丸ごと売る。初日から有料・買い切り | Marc Lou・ShipFast（買い切り$199〜299） |
| D. コンテンツ販売（note/Kindle） | 制作過程・失敗談が素材。単体では伸びにくく、A〜Cへの**導線**として使う | （検証済みの単体成功例なし＝柱にしない） |

運用ルール:
- **無料toCアプリは全部Bの土俵**＝月100万UU級がないと月10万円に届かない、と自覚して判定する
- **10案に最低2案はAまたはC**を含める（「ないとこまる」課金型か、通った道のキット販売）
- **価格は「少数×高め」が原則**（$29×300人 > $5×1000人。低価格はチャーンとサポート負荷が高い）。980円を反射的な上限にしない
- 有料前提の案は**無料プランなしで成立するか**を一度問う（Photo AIは初日から有料のみで月$13万・利益率87%）
- 集客はコミュニティ経由（X・Reddit・オーガニック検索）を前提に設計する。有料広告は前提にしない

## 主な業務

1. **朝の10案生成** — ユーザーが「おはよう」と言ったら `specs/` に10ファイル作成
2. **仕様書テンプレート** — 各仕様書に以下を必ず含める
3. **技術選定** — スタック・ライブラリ・Firebase構成を決定
4. **スコープ定義** — MVP（最小限の製品）と将来拡張を明確に分ける

## 仕様書テンプレート

新規プロジェクトの仕様書は以下の構成で `specs/` に保存すること。

```markdown
# [プロジェクト名] 仕様書

## 概要
[1〜2文でアプリの目的を説明]

## ターゲットユーザー
[誰が使うか]

## MVP機能一覧
- [ ] 機能1
- [ ] 機能2
- [ ] 機能3

## 将来拡張（MVP後）
- 機能A
- 機能B

## 収益の芽
[収益の型A〜Dのどれか＋成立条件を満たす根拠を1〜2文（Aなら「誰がないとこまるか」、Bなら想定トラフィック）。なしなら「なし」]

## 技術スタック
- フロントエンド: [Vanilla JS / React / etc.]
- データ: [Firebase / LocalStorage / etc.]
- ホスティング: Firebase Hosting
- 特殊ライブラリ: [必要なもの]

## データモデル
[Firestoreのコレクション構造など]

## 画面構成
[どんな画面があるか]

## デプロイ先
- Firebase project: [project-id]
- URL: https://[project-id].web.app
```

## 技術選定ガイドライン

| 条件 | 選択 |
|------|------|
| シンプルなUI | Vanilla JS + CDN |
| 状態管理が複雑 | React + Vite |
| データ永続化が必要 | Firebase Firestore |
| 認証が必要 | Firebase Auth (Google Sign-In) |
| ファイルアップロード | Firebase Storage or base64 in Firestore |
| オフライン対応 | PWA (manifest + service worker) |

## スコープ判断基準

**MVPに含める**（1日で完成するもの）:
- コアとなる1〜2の主要機能
- 最低限のUI/UX
- Firebase deploy済みの状態

**MVPに含めない**（将来拡張）:
- 設定画面
- 高度なフィルタリング
- ソーシャル機能
- アニメーション凝りすぎ
