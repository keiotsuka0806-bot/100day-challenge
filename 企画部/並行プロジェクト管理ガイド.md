# 並行プロジェクト管理ガイド

## 前提

- **基本ペース**: 1日1プロジェクト新規立ち上げ
- **同時進行**: 面白い企画は過去プロジェクトを改良・改善することで並行進行可能

## プロジェクトステータス定義

### 開発中
- 新規立ち上げプロジェクト（最大1日1個）
- MVP実装に注力

### QA待ち
- MVP実装が終わり、QA部のレビュー待ち
- 重大バグ・セキュリティ・モバイル表示の確認前

### デプロイ済み
- 本番URLは発行済み
- まだGenius Councilの3ループ改善前
- この状態では「友達に共有完了」と言わない

### デプロイ後改善中
- Genius Councilが実行済み
- Loop 1 / Loop 2 / Loop 3 の改善チケットを処理中
- 必要に応じて再デプロイを行う

### 共有可能
- 本番URLがHTTP 200
- QA通過済み
- Genius Councilの3ループ改善または明確な見送り判断が完了
- 友達にURLを渡してよい

### 改善中（並行進行）
- 前日以前の「面白い企画」
- 機能追加・バグ修正・UX改善
- 1日30分〜1時間程度の短時間作業

### 稼働中（本番）
- 共有可能化済み
- 継続的にユーザーが使用可能
- 定期的にメンテナンス

### 休止中
- スコープ外・技術検証済み
- 後日リバイバルの可能性

## 実装フロー — 新規 vs 並行改善

### 新規プロジェクト（優先度 HIGH）
```
朝: 「おはよう」で10案確認
　→「今日は[name]を作りたい」で開発部へ
午前〜夕方: MVP実装
夕方: QA部でレビュー
夜: 運用部がdeploy
deploy直後: Genius Councilで3ループ改善
```

### 並行改善（優先度 MEDIUM）
```
前日までのプロジェクト
「[昨日のproject]を改善したい」と指示
→ 開発部が短時間で機能追加・修正
→ QA部でテスト
→ デプロイ
→ Genius Councilで3ループ改善
```

## デプロイ後 Genius Council

新規・並行改善を問わず、デプロイされたプロジェクトは必ず Genius Council を通す。

```bash
cd /Users/kei/dev/100day-challenge
node 運用部/scripts/post-deploy-genius-cycle.mjs --project 開発部/[project-name] --url [production-url]
```

Genius Council はプロジェクト種別を自動判定し、ゲームならプロゲーマー集団、AIアプリならAIプロダクト集団、写真/共有アプリなら写真・SNS集団など、適切なプロ集団による3ループ改善を要求する。

この工程が終わるまで、プロジェクトステータスは「稼働中（本番）」ではなく「デプロイ後改善中」として扱う。

## 並行進行の判断基準

**以下の場合は並行改善を検討**:
- 前日のプロジェクトで「あったら便利」な機能がある
- ユーザーフィードバック（想定）がある
- 技術的に興味深い発展がある

**以下の場合は スキップ（新規優先）**:
- 新規プロジェクトに時間がかかりそう
- 改善に30分以上かかる見込み
- 新規企画の方が面白い

## ディレクトリ構成例

```
開発部/
  whisky-note/          ← 稼働中（本番）
  mood-forecast/        ← 共有可能
  pomodoro/             ← 今日新規立ち上げ（開発中）
  streak-board/         ← 改善予定（改善中）
  typing-zen/           ← アイデア段階（休止）
```

## CLAUDE.md での標記

各プロジェクトの `CLAUDE.md` 冒頭に以下のステータスを記載する:

```markdown
# [プロジェクト名]

**ステータス**: 開発中 / QA待ち / デプロイ済み / デプロイ後改善中 / 共有可能 / 改善中 / 稼働中 / 休止
**最終更新**: YYYY-MM-DD HH:MM
```

## 複数エージェント（部署）の並行進行ルール

**同時実行可能**:
- 新規企画の企画部 ↔ 前日プロジェクトの改善（開発部）
- 開発部（新規） ↔ QA部（別プロジェクト）

**シーケンシャル（順番待ち）**:
- 企画部 → 開発部 → QA部 → 運用部（同一プロジェクトは順番）
