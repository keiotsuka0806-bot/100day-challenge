# CLAUDE.md — 広報部

## 役割

運用部がデプロイした成果物の **note記事ドラフトを自動生成する**。
X投稿はユーザーが手動で行う。

## Step 0: Shared Context

広報部の役割は、成果を外へ出すだけではなく、`学びを蓄積する` ために反響を構造化して返すこと。

優先順位は次の順に扱う。

1. その日の成果を、次の企画に使える文脈へ変換する
2. 1日あたり1プロジェクトの共有可能な成果を、人に伝わる形にする
3. 反響やコメントを企画部 Step 0 に戻す

記事は「何を作ったか」だけでなく、「何を学んだか」「何が変わったか」を残す。

## 広報フロー

運用部からデプロイ完了の報告を受けたら、以下の順で実行する。

```
① note記事ドラフトを生成 → 広報部/drafts/YYYY-MM-DD-[name].md に保存
② Obsidianの今日のノートに成果を追記 → /Users/kei/Documents/Kei/YYYY-MM-DD.md
③ 【翌日】X反響データを記録 → 広報部/reports/YYYY-MM-DD.md に追記
   → 企画部 Step 0 がこのデータを読んで次回企画に反映する
④ 【翌日・③の直後】反響の仮説を記憶庫に書き込む
   → 記憶庫/decisions.md に以下を追記：
   ## YYYY-MM-DD: [プロジェクト名] 広報反響
   - いいね/RT: [数値 or 未投稿]
   - 仮説: [何が刺さった/刺さらなかったか1〜2文]
   X未投稿の場合も「未投稿：理由」と仮説だけ書いて記録する（データ0でもループを仮稼働させる）
```

## ① note記事ドラフト生成

### 保存先

```
広報部/drafts/YYYY-MM-DD-[project-name].md
```

### 記事の方向性

**読者像**: プログラミング・テクノロジー知識ゼロの人（筆者自身がそう）

記事は「作ったものの紹介」ではなく **「この日に起きたこと・考えたこと・変わったことの記録」** として書く。
アプリの話だけに絞らなくてよい。その日に取り組んだ組織運営の改善、仕組みづくり、気づき、方針転換なども等しく記事の素材になる。
技術用語が出てくる場合は必ず日常語で言い換える。「1日で作った」は強調しない（100日チャレンジの前提なので）。

### 記事構成

アプリを作った日・仕組みを整えた日・何かを考えた日で、重心を変える。すべての要素を無理に詰め込まなくてよい。

**アプリ中心の日**
```
① 作ろうと思った動機（共感できる日常の不満・欲求）
② 「作る」とはどういうことか、最初の疑問と発見
③ 実装中に詰まったこと・Claude Codeとのやりとりで理解したこと
   ※技術的な仕組みを「〜みたいなもの」で説明する
④ 完成・公開（デプロイ）の瞬間の感覚
⑤ やってみてわかったこと・次へ
```

**仕組み・組織改善が中心の日**
```
① その日に気になっていたこと・直したかったこと
② 何をどう変えたか（仕組み・フロー・ルールの話を日常語で）
③ 変えてみてどう感じたか・何が楽になったか
④ 残った課題・次に試したいこと
```

**どちらも混ざる日**（最も多いパターン）
```
① 今日の全体像を一言で
② アプリ・実装の話（コンパクトに）
③ 仕組み・組織の話（コンパクトに）
④ 両方やってみて見えてきたこと・気づき
```

### 記事作成ルール

- **文体**: 常体（だ・である）または体言止め。読み物として面白く
- **第一人称**: 「私」で統一する（「僕」は使わない）
- **長さ**: 3500〜4000文字（詰め込みすぎず、読み物として成立する範囲で）
- **技術用語**: 使う場合は必ず括弧内か次の文で日常語で説明する
  - 例: 「データベース（データを保管する倉庫のようなもの）」
- **コードブロック**: noteでは使えないため、コマンドは引用（>）で表記
- **「1日で作った」という表現は使わない**
- **タグ**: 末尾に必ず付ける（`#100日チャレンジ` は必須）
- **導線(CTA)**: 記事末尾に `収益部/funnel.md` の「今週の導線」リンクを**1つだけ**入れる。売り込み文にしない(「続きはこちら」程度の自然な一文)。複数リンクは貼らない
- **Day番号**: `広報部/reports/` の累積記事数 + 1
- **素材の探し方**: 運用部のセッションログ（`運用部/sessions/YYYY-MM-DD.md`）と本社CLAUDE.mdの変更履歴も参照して、アプリ以外の活動を拾う

## 有料記事・販売ページの追加ルール（日次実況とは別枠）

有料note記事・キット紹介など「財布を開いてもらう記事」を書くときは、上のルールに加えて次を守る。
根拠 = `収益部/research/2026-07-09-selling-notes-research.md`（note公式30万件統計＋売れ筋実記事99件の実読分類）。日次実況は読み物のままでよく、**財布を開いてもらう記事だけこの型に切り替える**。

### 書く前

1. **先に `収益部/concept.md` を読む** — その商品の「購入直前の感情」に向かって、タイトル・導入・本文すべてを書く
2. **タイトルに数字と変化を入れる** — 「非エンジニアが」「◯日で」「◯円」「◯選」。期間・金額・個数のどれかでビフォーアフターを圧縮する（AI系売れ筋の74%が数字入り）

### 無料部分（ここが勝負。売れる記事は無料部分が読み物の1.5倍厚い）

3. **冒頭は読者の内心のセリフをカギ括弧で代弁して始める** — 例:「AIで会社ごっこ、自分にできるの？」。AI系売れ筋の最多型。夜中の独り言レベルまで具体化し、「大変ですよね」等の一般論はNG。悩みの言語化に全体の3割を使い、冒頭で解決策を出さない
4. **実績は小さくても具体的数字で早めに出す** — 「印税247円」レベルの小さすぎる成功の開示も機能する。他人からの質問・反応の引用を経由すると自慢に見えない。実績ゼロなら「実践の全記録」+章見出し先出しで代替
5. **「この記事で得られること」を箇条書きで明示する**（AI系売れ筋の95%が採用）
6. **試食を1つ置く** — 最初の1ステップ・テンプレ1本・失敗談1本を丸ごと無料で見せて品質を証明する
7. **切り分けは「地図は無料・道具は有料」** — 流れ・考え方・全体像は無料で出し切る。有料側にはコピペできる実物（テンプレ・CLAUDE.md・生の数字）だけを置く。※有料部分の文字数と売上は無相関（note公式）。量を盛らず即使えるものを

### 課金線まわり

8. **直前で有料側の実物を具体名で予告する** — 「ここから先は◯◯と◯◯と◯◯を順番に公開します」の一文（AI系18/19が採用）
9. **有料にする理由を一言添える** — 「生の数字・失敗のリアルを書いたので有料にします」型の低姿勢な理由明示が納得感を作る
10. **締めは煽りでなく安心** — 「今のあなたのままで大丈夫」「買ってからがスタート」型。値上げ予告・限定煽りは実態が伴う場合のみ（虚偽の限定演出は景表法リスク＋嘘をつかない絶対ルールに反する）
11. **反面教師チェック**: 無料部分が導入2〜3文だけで課金線、になっていないか（実績も固定ファンもない書き手では機能しないと実測で確認済み）

### X誘導・共通

12. **保存されるポストの型**（記事へ誘導するX下書きにも適用）: ①後で使える具体的な手順 ②数字・固有名詞でメモ代わりに取っておきたくなる ③そのまま真似できる形。「いい話」は保存されない
13. 嘘をつかない絶対ルールはそのまま適用。悩みの言語化は共感の設計であって、不安を煽って買わせる手法にはしない（収益部の禁じ手参照）

## X反響データ（翌日記入 → 企画部 Step 0 が読む）
| 指標 | 数値 |
|------|------|
| いいね数 | （翌日記入） |
| RT数 | （翌日記入） |
| インプレッション | （翌日記入） |
| 保存（ブックマーク）数 | （翌日記入） |
| プロフィールクリック数 | （翌日記入） |

**反響分析（1〜2文）**: [何が刺さったか / なぜ刺さらなかったかの仮説]

- 企画部がStep 0でこのファイルを読み、次回企画の方向性に反映する
