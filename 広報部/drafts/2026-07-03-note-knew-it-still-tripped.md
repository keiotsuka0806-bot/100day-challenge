# タイトル案（3つ）

1. 「メモに書いた教訓を、また踏んだ日」──知っているのに直せない、を仕組みで直す
2. AIと会社をやって34日、いちばん多い失敗は「学習済みの失敗」だった
3. 教訓は"読むもの"にした瞬間、二度と効かなくなる

## リード（無料部分の冒頭）

うちの会社には「記憶庫」というフォルダがある。ハマった罠、うまくいったやり方を、AIが毎日書き溜めていく場所だ。半年で立派な教訓集になった。
なのに昨日、私はそこに書いてあるはずの罠を、また踏んだ。しかも今回が初めてじゃない。同じ罠を、新しいアプリを作るたびに踏んでいる。
「知っている」のに「直らない」。この気持ち悪さの正体と、たった一つの直し方の話をしたい。

## 本文構成

### 1. 「画面が真っ白」という毎度の罠
- 作ったアプリを更新しても、利用者の画面には古いままの中身が出続ける不具合（技術的には「Service Workerのキャッシュ優先」という設定が原因＝ブラウザが古いファイルを掴み続ける）
- これは半年前に一度どハマりして、記憶庫にちゃんと「次からはこう作れ」と書いてある
- なのに昨日、新しいアプリ「衝動ブレーキ」でまた踏んで、公開してから慌てて直した

[Keiの言葉: ここに「また踏んだと気づいた瞬間の、脱力とちょっとした自己嫌悪」を1〜3文。当時の実感で]

### 2. なぜ「書いてあるのに」踏むのか
- 教訓は記憶庫に"保管"されていた。でも新しいアプリを作り始める瞬間、私はそれを"読みに行かなかった"
- 知識は、必要な瞬間に目の前に無ければ存在しないのと同じ
- Karpathyというフロンティアの研究者が最近こう言っている。「RAG（必要なとき取り出す記憶）より、compile（最初から編み込まれた記憶）だ」と。取り出す前提の知識は、取り出し忘れた瞬間に無力になる

[Keiの言葉: ここに「専門用語は分からないけど、"探さないと使えない知識は無いのと同じ"は生活実感として刺さった」的な、非エンジニアならではの受け止めを1〜2文]

### 3. 直し方は「知る」じゃなく「既定にする」
- 昨日の結論：教訓を増やすのをやめる。代わりに「新しいアプリの雛形」そのものを、最初から正しい設定で作っておく
- つまり、正しいやり方を"思い出す"必要をなくす。雛形をコピーした時点で、もう罠が埋まっている
- これは料理でいえば「塩を入れ忘れないよう気をつける」のではなく「塩を最初から計量カップに入れておく」に近い

[Keiの言葉: ここに「気をつける、で直そうとするのはもうやめた」という決意めいた一言。仕事や暮らしの他の場面にも通じる実感があれば]

### 4. これは会社経営の話でもある
- 「マニュアルに書いてある」は、現場で開かれなければゼロ点
- 個人の努力（気をつける・覚えておく）で回すのをやめ、仕組み（既定・雛形）に肩代わりさせる
- 非エンジニアの自分がAIと会社をやってみて、いちばん効いた改善はいつも「頑張る」の反対側にあった
