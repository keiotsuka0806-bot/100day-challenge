# タイトル案（3つ。数字や具体を入れる）
1. 「直したのに直らない」——非エンジニアの私を3時間ハマらせた幽霊バグの正体
2. コードは新しいのに、画面は古い。AIと私が見落としていた1行
3. デプロイ＝公開、じゃなかった。Service Workerという見えない関所の話

## リード（無料部分の冒頭。3〜4文で「読む理由」を作る）
「入れない不具合」を直した。AIもOKと言った。なのに、ブラウザで開くとまだ直っていない。コードを取り出して確かめると、たしかに新しいコードが返ってくる——でも画面は古いまま。プログラミングを正式に学んだことのない私が、この「幽霊」と数時間にらみ合って学んだのは、技術の知識そのものより「直ったかどうかを、どこで確かめるか」という話だった。

## 本文構成

### 1. 事件：直したはずのバグが、画面にだけ残っていた
- 「あいだ」というアプリで「入れない」不具合を修正
- AIに確認させてもOK。デプロイも成功
- でもスマホ/ブラウザで開くと、古い挙動のまま
- [Keiの言葉: ここに当時の気持ち——「自分のミス？AIのミス？それとも何かが壊れてる？」という混乱を1〜3文]

### 2. 犯人：Service Worker（見えない関所）
- アプリには「次もすぐ開けるように」中身を手元に保存しておく仕組みがある（PWA/Service Worker）
- これが「まず手元の古い版を見せる（cache-first）」設定だと、新しい版が届いても古いものを掴み続ける
- curl（コードを直接取り出す道具）では新コード、ブラウザは旧コード——この食い違いが「幽霊」の正体

### 3. 学び：非エンジニアこそ「確認する場所」を間違えると無限に溶ける
- 私は「コードを直す」までを仕事だと思っていた。本当は「利用者の画面で直っている」までが仕事だった
- 対策はシンプル：更新が多いアプリは「まず新しい版を取りに行く（network-first）」に変える
- そして同じ轍を踏まないよう、これを会社の「記憶庫」に1行で残した。次の私（とAI）はもうここで溶けない

### 4. AIと会社をやる私にとっての意味
- AIは「コードは正しい」と言える。でも「あなたの利用者が今どう見えているか」は見ていない
- だから人間の私の仕事は、コードの正しさではなく「現実での体験」を確かめること
- [Keiの言葉: ここに——AIに任せる範囲と、自分が最後に立つ場所をどう分けるか、についての実感を1〜3文]
