# 「この名言、本物ですか？」を裁判形式で判定するゲームを作った。

有名な名言の引用を見るとき、本当にその人が言ったのか確認することはほとんどない。

「マザー・テレサの言葉」として広まっているものが、実はその人の言葉ではなかった、という話がインターネット上には溢れている。でもシェアされるとき、出典は曖昧なままだ。「名言だからきっと誰かが言ったはず」という空気で流通している。

だったらそこをゲームにしよう、と思った。名言を見せて「本物か偽物か」を判定するゲーム——名言法廷、Quote Courtとして作った。

---

## Claude APIで名言のケースを生成する

Quote Courtのゲーム内容は10件の「事件」を解くことだ。各事件では一つの名言が提示されて、「これはその人物が本当に言った言葉か」「誤引用・フェイクか」を判定する。正解数・確信度・コンボによって最終スコアが決まる。

肝心の「問題」はClaude APIを使って生成した。「本物の名言5件・誤引用または架空の名言5件を、人物・時代・背景とともにセットで生成して」という指示を出す。Claude APIは毎回違う問題を返してくれるので、プレイするたびに別の内容になる。

問題を固定のデータとして自分で作ろうとすれば、何百件も用意しなければならない。でもAIに生成させれば、理論上は無限のバリエーションが生まれる。「コンテンツの生成をAIに任せる」という発想は、このプロジェクトで初めて本格的に使った。

## APIキーをどうやって隠すか

ここで初めて「セキュリティ」を真剣に考える場面が来た。

WhiskyNoteやMoodForecastはFirebaseの機能を使っていたので、APIキーの扱いはFirebaseの設定で何とかなった。でもClaude APIは違う。ブラウザのJavaScriptの中にClaude APIキーを書いてしまうと、サイトを開けば誰でもそのキーを見られる状態になる。キーが漏れると、他人が自分のアカウントでAIを使い放題になってしまう。

解決策はシンプルで、「APIキーをブラウザには置かず、自分のサーバーに隠す」というものだ。

```
ブラウザ → 自分のサーバー → Claude API
```

ブラウザは自分のサーバーにリクエストを送り、サーバーがClaude APIを呼んで結果を返す。キーはサーバーの中にだけある。

これを実現するための仕組みとして、Firebase FunctionsとVercel Functionsの両方を試した。どちらも「サーバー側で動くJavaScript」を書ける仕組みだ。Firebase FunctionsはGoogleのクラウドで動き、Vercel FunctionsはVercelのクラウドで動く。構造は似ているが、設定の書き方や動作確認の方法が違う。

両方を実装してみたことで、「どちらが自分のプロジェクトに合うか」を比較できた。一種の社内競合だ。

## ChatGPTとの対話から学んだこと

この日、Claude Codeだけでなく、ChatGPTとも対話しながら作業した。

気づいたのは「質問の仕方によって返ってくる答えの質が変わる」ということだ。Claude Codeに「このコードを直して」と言うより「なぜこのコードはエラーになるのか教えて、それから修正案を出して」と言う方が、より理解できる答えが来る。

「具体的にどんな質問を投げかければいいか」を教えてもらうことで、学びの速度が上がる気がした——Obsidianのメモにその日こう書き残している。

AIは答えを出す機械ではなく、「どう問うか」を一緒に考えてくれる対話相手だ。Quality of Question（質問の質）がQuality of Answer（回答の質）を決める。この感覚を持てたのは、この日の収穫だった。

## Wikipediaから人物の画像を取得する

Quote Courtでは、名言の人物がゲーム画面に表示される。架空の名言を「本物らしく」見せるためにも、人物の顔写真や情報があると説得力が増す。

そこで使ったのがWikipedia REST APIだ。人物名を送ると、Wikipediaのページにある情報や画像URLを返してくれる。マザー・テレサを検索すれば、Wikipediaに載っている顔写真のURLが返ってくる。

これもまた「外のデータを借りる」パターンだ。NaniTaberuで楽天レシピAPIを使ったのと同じ発想で、今度は百科事典のデータを借りた。何かを作るたびに、「このデータを自分で用意する必要があるか、どこかから借りられるか」を考えるようになった。

## Web Audio APIで効果音を鳴らす

ゲームの演出として、正解・不正解の効果音も加えた。

音を鳴らすために「Web Audio API」というブラウザ標準の機能を使った。音声ファイルを用意しなくても、JavaScriptのコードだけで音の波形を生成して鳴らせる。「ピー」という電子音から「ドン」という低音まで、コードで音を描くような作業だった。

WhiskyNoteを作ったとき「テキストファイル3つで動くアプリができる」と驚いた。その驚きの先に「音もコードで作れる」という発見があった。ブラウザの中には、開発者が使える道具が思っていたより多くある。

## 日々「もう一本」を同時進行させている

この日も、Quote Courtを作りながら別のプロジェクトの構想を進めていた。ワイン産地を舞台にしたボードゲーム「VinRoute」の産地マップを、15産地から35産地に拡張する設計だ。

毎日1プロジェクトを作ると決めたが、「今日のメイン」と「明日以降の仕込み」が常に並行して動いている。完成品を作る手と、次の構想を考える手が、同時に動いている状態だ。

これが100日チャレンジの実態だと思う。1日1個というのは、1日に1個しか動かさないという意味ではなく、1日1個を「完成させる」ことを最優先にするという意味だ。
