# VinRoute 本格実装 — Claude Code への指示書

> このファイルは Claude Code にそのまま渡す実装指示書。以下の2つの文書と1つのプロトタイプとセットで使う：
> 1. **VinRoute_重量級_設計案.md** — ゲーム仕様の正（v1.0・プロトタイプ検証済み）
> 2. **VinRoute_データ分離設計.md** — アーキテクチャの正（country.json スキーマ・コード構造）
> 3. **vinroute_prototype.html** — 動くプロトタイプ（全システム実装済み・挙動のリファレンス）

---

## 最初に読むこと（あなた＝Claude Code への依頼の全体像）

フランスワイン収集すごろく「VinRoute」を、プロトタイプから本格実装に作り直してほしい。

**最重要の方針：産地・地形・ランドマーク・都市・ルート・クイズなどの国固有データを country.json（まず france.json）に分離し、エンジンとUIは国に依存しないコードにすること。** 将来イタリア・スペインを「JSONを足すだけ」で追加できる構造が、この実装の存在理由。単なる移植ではなく、データ分離した再構築である。

仕様の疑問はまず設計書を見る。設計書に無い細部はプロトタイプの挙動が正。両方に無いものだけ質問してほしい。

## 技術スタック（指定）
- フロントエンドのみで動く構成（プロトタイプ同様、サーバー不要で起動できること）
- Vanilla JS + Canvas を基本とする（プロトタイプの資産を活かす）。ビルドツールは Vite 程度の軽いもの可
- モジュール分割は データ分離設計.md §4 のディレクトリ構成に従う
- テスト：engine/ は DOM 非依存にし、Node（vitest等）で自動テストする

## 実装ステップ（この順で。各ステップで動作確認してから次へ）

### Step 0：データ抽出
- prototype.html から FR_LATLON / CORSICA_LATLON / FR_BBOX / REGION_LL / LINKS / REGION_TYPE / TYPE_SETS / GEO_SETS / QUIZ を抽出し、データ分離設計.md §2 のスキーマで **france.json** を作る。
- 産地（20）と都市（10）を regions / cities に分離。paris は cities に置き isStart:true。
- 抽出結果の件数を検証：産地20・都市10・links全件・タイプ4種。

### Step 1：盤面エンジン（engine/board.js）
- データ分離設計.md §3 のパイプラインを実装（投影→配置→ルート→メッシュ→接続保証→マス種配布）。
- プロトタイプの buildBoard / routeStaircase / meshConnect / pruneOverBranched / ensureRegionConnectivity のロジックを国非依存化して移植。乱数は seed 制御。
- **自動テスト必須**：全マス到達可能・全産地/都市の接続≥2・行き止まり0・密度バランス（3分割帯の偏り）。

### Step 2：コアループ（engine/state.js + 最小UI）
- ターン/シーズン進行（7ターン×5シーズン）、サイコロ（**出目を先に確定してから表示**）、移動・分岐選択、産地での建設。
- モーダル待ち合わせは「完了コールバックでターン再開」に統一（設計書§14の学び1。デッドロック厳禁）。

### Step 3：経済（engine/economy.js）
- 資源3種・資産3種・生産・再投資（Lv3まで）・売る/持つ・試飲料（最高評価のみ徴収）。
- 総資産 = value + money + 資産価値(5/8/12)。

### Step 4：評論家・ヴィンテージ・注目産地（engine/critic.js / vintage.js）
- 設計書§6.1の効果テーブル（軽/中/大惨事2系統）をデータ定義で。なすりつけ（同マス＋すれ違い・1手番1回）。
- パーカーポイント（設計書§5.9の計算式そのまま）。注目産地レース＋分岐の近道矢印。
- 評論家モーダルはトークン管理（古いタイマーが新しい表示を閉じない）。

### Step 5：コレクション（engine/collections.js）
- **保有ベース**（設計書§5.5.3）：タイプ/産地/ヴィンテージは手元の wines のみで判定。地理セットだけ畑ベース。
- 判定タイミング：建設後・生産後・購入後・落札後。二重取得防止フラグ。
- ボーナスは必ず addBonus(p, cat, amt) 経由で bonusLog に記録（エンディング内訳用）。

### Step 6：都市マーケット＋オークション
- 買う/卸す(70%)/競売(90%開始・全員入札・NPCは自動判断)。設計書§5.10の確定仕様通り。

### Step 7：カード＋クイズ
- カード約19種を CARD_DEFS 形式のデータ定義で移植（50種へ拡張できる構造）。ぴったりカードは目選択UIまで含む。
- クイズはデータ外部化（quiz_france.json）。まずプロトタイプの問題を移植し、拡充できる構造に。

### Step 8：エンディング＋共有カード
- 設計書§11の流れ（内訳→特別賞→順位→優勝者→共有カード）。ゆっくりスクロール。
- 戦績保存（localStorage）：順位・内訳・日時を保存し一覧表示。

### Step 9：NPC AI強化
- BFSで目的地選択、建設優先度、オークション入札上限、コレクション狙い。プロトタイプの簡易AIを土台に。

### Step 10：構造の証明
- italy.json のスタブ（10産地程度・実在のイタリア産地）を作り、**エンジン変更ゼロ**で盤面生成・プレイできることを確認。これが合格基準（データ分離設計.md §7）。

## してはいけないこと
- 国名・産地名・座標・クイズ内容をエンジンコードにハードコードすること
- モーダル表示中にターンが進む実装（プロトタイプで2件のデッドロックの原因）
- サイコロの表示値と実移動数が別々に決まる実装
- 全プレイヤーに一律で同じだけ効くペナルティの追加（順位が変わらず無意味）

## 各ステップの完了条件
- 自動テストがパスすること（engine はNodeテスト、盤面は検証3項目）
- 該当機能をブラウザで実際に操作して確認したこと
- 前のステップの機能が壊れていないこと（回帰確認）
