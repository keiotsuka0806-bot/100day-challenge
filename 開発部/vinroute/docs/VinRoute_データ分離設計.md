# VinRoute データ分離設計（全世界展開アーキテクチャ）

> 目的：**フランス固有のデータを country データ（france.json）に切り出し、エンジン・UIを国非依存にする。** イタリア・スペイン・世界周遊は「国データを追加するだけ」で実現できる構造にする。
>
> 位置づけ：設計書 v1.0（VinRoute_重量級_設計案.md）とセットで Claude Code に渡す。設計書が「何を作るか」、本書が「どう構造化するか」。

---

## 1. 分離の原則

**「その国でしか成り立たない情報」だけを国データに置く。ルール・数値・アルゴリズムはエンジンに置く。**

| 置き場所 | 含むもの | 例 |
|---|---|---|
| **国データ（france.json）** | 地理・産地・都市・道・地形・文化的ラベル・クイズ | 国の輪郭座標、ボルドーの緯度経度、産地間ルート、ワインタイプ、クイズ問題 |
| **エンジン（国非依存）** | ルール・数値バランス・アルゴリズム・状態管理 | ターン/シーズン進行、資源・建設・生産、評論家、オークション、コレクション判定ロジック、盤面生成アルゴリズム、BFS |
| **UIレイヤー（国非依存）** | 描画・モーダル・演出 | Canvas描画、HUD、建設メニュー、エンディング |

判断基準：「イタリア版を作るとき、このコード/データは書き換わるか?」→ 書き換わるなら国データ、書き換わらないならエンジン。

## 2. country.json スキーマ（v1）

```jsonc
{
  "id": "france",
  "name": "フランス",
  "nameEn": "France",
  "currencySymbol": "💰",          // 将来の国別通貨演出用（現状共通）
  
  // ── 地理 ──────────────────────────
  "geo": {
    "bbox": { "minLon": -4.9, "maxLon": 9.8, "minLat": 41.2, "maxLat": 51.2 },
    "outlines": [                    // 本土＋島。複数ポリゴン対応
      { "name": "mainland", "latlon": [[51.05,2.55],[50.95,1.85], ...] },
      { "name": "corsica",  "latlon": [[43.0,9.4], ...] }
    ],
    "terrain": [                     // 山脈・河川など装飾地形（任意）
      { "type": "mountain", "latlon": [45.8,6.9], "label": "アルプス" }
    ],
    "landmarks": [                   // 観光ランドマーク（描画用）
      { "type": "eiffel", "latlon": [48.86,2.29] },
      { "type": "abbey",  "latlon": [47.0,4.8] }
    ]
  },

  // ── 産地・都市 ──────────────────────
  "regions": {
    "bordeaux": {
      "name": "ボルドー", "nameEn": "Bordeaux",
      "group": "red",                // red | white | sparkle | rose
      "type": "赤",                  // コレクション判定用タイプ（表示名）
      "value": 35,                   // 産地の基礎価値
      "lat": 44.84, "lon": -0.58,
      "mark": "winecity"             // 描画マーカー種
    },
    // ... 全20産地
  },
  "cities": {
    "paris": { "name": "パリ", "lat": 48.85, "lon": 2.35, "mark": "eiffel", "isStart": true },
    "lyon":  { "name": "リヨン", "lat": 45.76, "lon": 4.84, "mark": "city" }
    // ... 全10都市。都市は全て市場機能を持つ（isStartは開始位置のみを意味する）
  },

  // ── ルート（産地・都市間の道） ─────────
  "links": [
    { "a": "paris", "b": "champ", "steps": 5 },
    { "a": "champ", "b": "alsace", "steps": 6 }
    // ... a/b は regions か cities のキー
  ],

  // ── セット定義 ──────────────────────
  "sets": {
    "geo": {                         // 地理セット（畑ベース）
      "北東": ["champ", "alsace"],
      "南西": ["bordeaux", "cognac", "cahors", "madiran", "gaillac"]
      // ...
    },
    "typeLabels": ["赤", "白", "泡", "ロゼ"]  // タイプコレクションの構成（国により変わりうる）
  },

  // ── クイズ（国テーマ） ─────────────────
  "quiz": [
    {
      "q": "ボジョレー・ヌーヴォーの解禁日は?",
      "choices": ["11月第3木曜", "12月第1金曜", "10月末日", "11月11日"],
      "answer": 0,
      "explain": "毎年11月の第3木曜日に世界同時解禁される。"
    }
    // ... 目標1000問。別ファイル分割可（quiz_france_01.json 等）
  ],

  // ── 国別フレーバー（演出テキスト・任意） ──
  "flavor": {
    "endingMemo": "パリから始まった旅。ボルドー、ブルゴーニュ…",
    "criticName": "辛口評論家"
  }
}
```

### スキーマの注意点
- **座標は必ず実在の緯度経度**で持つ。盤面座標への変換（bbox正規化→グリッドスナップ）はエンジンの仕事。
- regions と cities を分ける（プロトタイプでは REGIONS に混在していた。分離時に整理する）。
- links の steps は「その区間の道中マス数の目安」。実際のマス生成はエンジンのグリッドアルゴリズムが行う。
- クイズは量が大きくなるため、国データ本体と分割ロードできる構造を推奨。

## 3. エンジンが国データから盤面を作る流れ

プロトタイプで実証済みのアルゴリズムをそのまま国非依存化する：

```
loadCountry(json)
  → projectAll()        … bbox正規化で latlon→論理座標(0-100)
  → placeNodes()        … 産地・都市をグリッドスナップ配置（重なり回避）
  → buildRoutes()       … links を階段状ルートで接続（交差点が分岐を生む）
  → meshConnect()       … 控えめな基本網(0.32) → 一本道に分岐追加 → 過剰分岐の間引き
  → ensureConnectivity()… 全産地・都市を最低2方向接続、全マス到達可能を保証
  → assignCellTypes()   … 道中マスに種類（カード/クイズ/市場/…）を決定的乱数で配布
```

**検証要件（ビルド時に必ず自動チェック）**：
1. 全マスが開始地点から到達可能（BFS）
2. 全産地・都市の接続数 ≥ 2（行き止まりなし）
3. 盤面を3分割した各帯のマス/分岐数の偏りが一定以内（密度バランス。プロトタイプの学び）

## 4. コード構造（Claude Code 実装時の推奨）

```
vinroute/
├── data/
│   ├── france.json          # 国データ v1
│   ├── quiz_france.json     # クイズ大量データ（分割）
│   └── (italy.json …)       # 将来追加
├── engine/
│   ├── board.js             # 盤面生成（§3のパイプライン）＋検証
│   ├── state.js             # ゲーム状態（プレイヤー・シーズン・資源）
│   ├── economy.js           # 建設・生産・売買・オークション・再投資
│   ├── collections.js       # 保有ベースコレクション＋地理セット判定
│   ├── critic.js            # 評論家（憑依・ダメージテーブル・なすりつけ）
│   ├── vintage.js           # ヴィンテージ・注目産地・パーカーポイント
│   ├── cards.js             # カード定義と効果
│   ├── npc.js               # NPC AI
│   └── config.js            # 数値バランス（CONFIG。国非依存）
├── ui/
│   ├── map.js               # Canvas描画（地形・道・マス・アバター・矢印）
│   ├── modals.js            # 建設/市場/競売/売却/評論家/クイズ 等
│   ├── hud.js               # HUD・シーズンバー・凡例
│   └── ending.js            # エンディング（内訳・共有カード）
└── main.js                  # 起動・国選択・ゲームループ
```

### 重要な設計ルール
1. **engine/ は DOM を触らない**（テスト可能性。プロトタイプのNodeスモークテストの教訓を活かし、エンジンは純粋ロジックにする）。
2. **UIとエンジンの境界はイベント/コールバック**。モーダルの「閉じるまでターンを止める」待ち合わせは、エンジン側が完了コールバックを受け取る形に統一（プロトタイプのデッドロック2件の教訓）。
3. **CONFIG（数値バランス）は国データに入れない**。ゲームルールは全国共通。国ごとに難易度を変えたくなったら country.json に override フィールドを足す（将来）。
4. **乱数は seed 制御**（盤面生成は決定的に。同じ国データ→同じ盤面。リプレイ・デバッグ可能性）。

## 5. 全世界展開のパス

### フェーズ1：フランス単国（本格実装の初期目標）
- france.json ＋ エンジンで、プロトタイプと同等のゲームを再現。

### フェーズ2：2国目（イタリア）で構造を証明
- italy.json を追加（バローロ、キャンティ、ブルネッロ…）。
- **エンジンのコード変更ゼロで動くこと**が構造の合格基準。
- タイプ構成が国で違う場合（例：イタリアに泡=フランチャコルタ、スプマンテ）も typeLabels で吸収できるか検証。

### フェーズ3：世界周遊モード
- 複数国を順に巡る（1国=Nシーズン→次の国へ）。
- 国間の持ち越し：資金・ワインコレクションは持ち越し、資産（畑等）は国に残る…等のルールは設計書§5.6の調整レバーと合わせて将来設計。

## 6. プロトタイプからの移行マッピング（何がどこへ行くか）

| プロトタイプの要素 | 移行先 |
|---|---|
| FR_LATLON / CORSICA_LATLON / FR_BBOX | france.json → geo.outlines / geo.bbox |
| REGION_LL（20産地＋10都市） | france.json → regions / cities に分割 |
| LINKS | france.json → links |
| REGION_TYPE / TYPE_SETS / GEO_SETS | france.json → regions[].type / sets.geo |
| QUIZ（5問） | quiz_france.json（1000問へ拡充） |
| CONFIG / CRITIC_EFFECTS / CARD_DEFS / SLOT_RES | engine/config.js・critic.js・cards.js（国非依存） |
| buildBoard / meshConnect / pruneOverBranched / ensureRegionConnectivity | engine/board.js（そのまま国非依存化） |
| ゲーム進行・経済・コレクション・評論家・エンディング | engine/ 各モジュール |
| Canvas描画・モーダル・HUD | ui/ 各モジュール |

---

## 7. 合格基準（この設計が成功したと言える条件）
1. france.json を読み込んでプロトタイプと同等のゲームが動く。
2. italy.json（最小10産地程度のスタブ）を追加して、**エンジン変更ゼロ**でイタリア盤面が生成・プレイできる。
3. 盤面検証（到達可能性・接続数・密度バランス）が自動テストで常時パスする。
4. engine/ が DOM なしの Node テストで検証できる。
