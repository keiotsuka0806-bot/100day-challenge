## Council 議論

- **SaaS PM**: 初回の価値到達（3問→公的保障カード）は30秒以内で成立している。弱いのは再訪。保険の見直しは一度きりではないのに、再訪時に前回の診断へ戻る導線がない。
- **業務改善コンサル/FP視点**: 判定エンジンの家族構成分岐に穴がある。「自営業・配偶者あり・子なし」は遺族年金がほぼゼロになる特殊ケースで、汎用文だと「遺族年金が月あたり約0円出ます」という誤解を招く出力になる。このケースはむしろ死亡保障の必要性が高い世帯で、逆方向の判定を返すべき。
- **アクセシビリティ専門家**: 選択肢のfocus-visible・16px入力・44pxタップ領域は担保済み。ラジオ選択のカード型UIはinput不可視だがフォーカスリングが出ることを確認。
- **QAエンジニア**: 0円入力・未入力エラー・localStorage破損退避・保存失敗の巻き戻しは実装済み。共有導線としてSHARE.md（友達に送る文面）が未作成。

## 3ループ改善チケット

### GC-L1: Loop 1: First-Use / Core Value

- **問題**: 初見の価値到達は3問→「あなたはすでに月◯円の保険に入っています」カードで成立（実測30秒以内・3操作）。ただし再訪ユーザーがホームから前回の結果に戻れず、毎回3問からやり直しに見える。
- **完了条件**: 保存データがある状態でホームを開いたとき、前回の診断（公的保障カード or 3色マップ）へ1タップで戻れる。
- **実装結果**: ホームに「前回の続きを見る」ボタンを追加（プロフィールのみ保存→公的保障カードへ／保険入力済み→3色マップへ。文言も状態で切替）。
- **確認方法**: 実ブラウザで前回データ保存済み状態のホームを開き、`前回の3色マップを見る`が表示・遷移することを実測。
- **判定**: Done

### GC-L2: Loop 2: Expert Quality / Domain Depth

- **問題**: 「自営業・配偶者あり・18歳以下の子なし」の死亡保険判定が汎用文に流れ、「遺族年金が月あたり約0円出ます」という不自然かつ誤解を招く文になる（遺族基礎は子が要件・遺族厚生は自営業に無い）。このケースは公的保障の空白であり、黄（上乗せ）でなく赤（穴を埋めている可能性）が正しい。
- **完了条件**: 遺族年金がほぼゼロになる世帯構成で、死亡保険の判定が赤（穴を埋めている可能性）となり、「約0円」のような誤解を招く文言が出ない。
- **実装結果**: judgePolicy()のdeath-termに遺族年金ゼロの専用分岐を追加。判定を「あなたの本当の穴を埋めている可能性」（赤）に変更し、「安易に削る対象ではない」と明示。質問リストも専用文に差し替え。
- **確認方法**: 実ブラウザで該当プロフィール（自営業・配偶者あり・子なし）を入力し、判定・文言・「約0円」が出ないことを実測。
- **判定**: Done

### GC-L3: Loop 3: Replay / Share / Operation

- **問題**: 共有の型（SHARE.md）が未作成。友達に渡す文面・スクショの瞬間の定義がない。
- **完了条件**: 友達に送れる紹介文・スクショの瞬間の定義・再訪導線・共有画像の書き出しが揃っている。
- **実装結果**: SHARE.md作成（短い版/きっかけ版の紹介文・スクショの瞬間=公的保障カード・「言わないこと」の設計思想）。PNGカード書き出し（公的保障カード/結果カード）と質問リストコピーは初期実装済み。再訪導線はGC-L1で対応。
- **確認方法**: SHARE.mdの文面でURLを渡し、追加説明なしで主要価値（公的保障カード）に到達できる構成であることをレビュー。
- **判定**: Done
