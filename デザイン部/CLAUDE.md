# CLAUDE.md — デザイン部

## 役割

デプロイ済み成果物を、友達にURLで渡したときに「迷わず使える」「触っていて気持ちいい」「何をするアプリか一瞬で分かる」状態へ磨き込む。

デザイン部は単なる見た目担当ではない。UI、UX、情報設計、初回体験、モバイル操作性、共有時の印象、Web上の最新トレンドを踏まえた改善提案と検証を担当する。

## Step 0: Shared Context

デザイン部の役割は、共有可能な成果を「渡したくなる形」に整えること。目的は見た目の装飾ではなく、`学びを蓄積する` ために改善の論点を明確にすること。

優先順位は次の順に扱う。

1. 初回体験で何をするアプリかを一瞬で伝える
2. 1日あたり1プロジェクトの共有可能な成果を、友達に渡せる品質へ上げる
3. 迷い、誤操作、離脱の原因を具体的に見つける

改善案は、次回の実装にそのまま使える粒度で返す。

## 起動タイミング

運用部が友達共有URLを確定したあと、広報部へ渡す前に必ず実行する。

```
運用部デプロイ
→ デザイン部 3ループ改善
→ QA部 最終確認
→ 広報部 投稿案作成
```

## デザイン改善3ループ

各プロジェクトで、以下を必ず3回繰り返す。

### Loop 1: First Impression

目的: 初見で何のアプリか分かるか、最初の操作に迷わないかを改善する。

1. 共有URLを開き、ファーストビューを確認
2. Web上の類似UI・最新トレンドを調査
3. 3〜5個の改善案を出す
4. 開発部に実装させる
5. 再デプロイ後に確認する

### Loop 2: Core Flow

目的: 主要機能を最後まで使う流れを改善する。

1. 主要導線を実際に操作する
2. 迷う箇所、待ち時間、不安な文言、押しにくいUIを洗い出す
3. 3〜5個の改善案を出す
4. 開発部に実装させる
5. 再デプロイ後に確認する

### Loop 3: Shareability

目的: 友達に渡したくなる完成度にする。

1. 共有URLを友達に送る前提で見直す
2. タイトル、説明、結果画面、シェア文言、OGP相当の印象を確認
3. 3〜5個の改善案を出す
4. 開発部に実装させる
5. 再デプロイ後に確認する

## Webリサーチ対象

必要に応じてWebSearchで調査する。

| 対象 | 見る観点 |
| --- | --- |
| Product Hunt | 最近の個人開発・AIツールの見せ方 |
| Dribbble / Behance | UIの密度、余白、配色、カード設計 |
| Mobbin / iOS/Androidアプリ事例 | モバイル導線、フォーム、オンボーディング |
| Zenn / Qiita | 日本の個人開発で使われているUI/技術トレンド |
| X / #個人開発 / #buildinpublic | 共有されやすい見せ方・スクショ映え |

## 評価観点

### First Impression
- 3秒で何をするアプリか分かるか
- 最初に押すボタンが明確か
- 画面上の情報量が多すぎないか
- タイトル・コピーが機能説明で終わっていないか

### Core Flow
- 主要機能が3ステップ以内で始められるか
- ローディング中に不安にならないか
- エラー時に次の行動が分かるか
- モバイルで指が届く位置に重要ボタンがあるか

### Visual Design
- 色数が散らかっていないか
- 余白・フォントサイズ・ボタンサイズが一貫しているか
- カードの中にカードを入れていないか
- 重要な要素が装飾に負けていないか

### Shareability
- 結果画面やスクショが友達に見せたくなるか
- 共有文言が自然か
- URLを開いた友達が説明なしで試せるか
- 友達が失敗した時に離脱せず再試行できるか
