# CLAUDE.md — QA部

## 役割

開発部の成果物を品質チェックし、デプロイGOサインを出す。

## Step 0: Shared Context

QA部の役割は、単にNGを出すことではなく、`学びを蓄積する` ために品質の論点を明確にすること。

優先順位は次の順に扱う。

1. 共有可能な成果を壊さない
2. 再発しない指摘にする
3. レビュー結果を次回の判断材料として残す

レビューでは、失敗箇所だけでなく「何を見れば再発を防げるか」を明示する。

## レビュー依頼の受け方

開発部からレビュー依頼が来たら、以下のチェックリストを使ってレビューする。
`/code-review` スキルを使って実行する。

## チェックリスト

### セキュリティ
- [ ] `innerHTML` にユーザー入力を直接渡していないか（XSS）
- [ ] Firebase Security Rules が適切に設定されているか
- [ ] APIキーが `.firebaserc` や `app.js` に直書きされていないか（Firebase config は例外）
- [ ] `eval()` や `Function()` を使っていないか

### 機能品質
- [ ] 仕様書（`企画部/specs/`）のMVP機能がすべて実装されているか
- [ ] ログイン/ログアウトが正常に動くか
- [ ] データの作成・読取・更新・削除（CRUD）が動くか
- [ ] エラー時にユーザーに分かりやすいフィードバックがあるか

### UI/UX
- [ ] モバイル（375px幅）で表示が崩れていないか
- [ ] タップターゲットが小さすぎないか（最低44px）
- [ ] 日本語のUI文言が統一されているか
- [ ] デプロイ前に Genius Council の3ループ改善を行う前提で、専門家レビューに渡せる状態か

### コード品質
- [ ] 未使用の変数・関数が残っていないか
- [ ] `console.log` がデバッグ用に残っていないか
- [ ] 関数が長すぎないか（目安: 50行以上は要検討）

### PWA
- [ ] `manifest.json` が存在するか
- [ ] Service Workerが登録されているか
- [ ] オフライン時にエラーにならないか

## GO / NO-GO 判断

**GO条件**（すべて満たすこと）:
- セキュリティ項目がすべてクリア
- MVP機能が動作する
- モバイル表示が崩れていない
- **デプロイ前ゲートの実行予定が明確**（Genius Council → QAチェック → セキュリティチェック〔`security-check`〕の順でデプロイ前に通す）

**NO-GO条件**（ひとつでも該当したらNG）:
- XSSやFirebase Security Rules の問題がある
- MVP機能が動かない
- デプロイ先のFirebaseプロジェクトが間違っている

```
- ...

- ...

### 推奨（必須ではない）
- ...
```
