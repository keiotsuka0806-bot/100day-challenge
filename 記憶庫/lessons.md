## 2026-06-24（朝会レトロ：git hygieneの"解消"は嘘だった）
- **[git hygiene／嘘をつかない]** 6/22の `fe0378a「git hygiene 解消」` と lessons の 6/22 エントリは**実態を先取りした誤記録**だった。コミットは `.gitignore` を編集しただけで `git rm --cached` をしておらず、**.gitignore は既に追跡済みのファイルを追跡解除しない**ため、8件のスキルカタログ＋2件の`.env*.example`は追跡されたまま・フックは6/24まで失敗継続していた。本日 `git rm --cached` で6ファイル（複製カタログ4本＋env2本）を実際に追跡解除し、hygiene が初めて exit=0「ok」になった。
- **[教訓]** 「追跡解除」は `.gitignore` 追加ではなく `git rm --cached` が本体。**コミットメッセージや記憶庫に"解消した"と書く前に、フックを再実行して exit=0 を目視確認する**（嘘をつかない原則）。`git ls-files | grep <pattern>` が空かどうかで追跡残を確かめられる。

## 2026-06-14
- **[PWA/Service Worker]** SWが**cache-first**(`caches.match(req) || fetch`)だと、デプロイしても利用者のブラウザは**古いapp.jsを掴み続け、修正が反映されない**(あいだで「入れない不具合を直したのに直らない」が再現。curlでは新コードが見えるのにブラウザは旧コードだった)。対策: 頻繁に更新するアプリはSWを**network-first**(`fetch().then(cacheに保存).catch(cacheにフォールバック)`)にする。SW更新時は`skipWaiting()`+`activate`で`clients.claim()`。利用者側は1〜2回のリロード/タブ開き直しで新SWが反映される。
- **[Vercel/モノレポ]** `開発部/[app]` に**自前の `.vercel` が無い**まま `vercel` を打つと、リポジトリに紐づく**別の既存プロジェクトを自動で拾って誤デプロイ**する(あいだのファイルがfood-scoreに飛んだ。2026-06-12の404事故と同根)。対策: ①新規アプリはリポジトリ外コピーから初回デプロイして専用プロジェクトを作る→生成された`.vercel`を本体フォルダにコピー ②リポジトリ直下に`.vercel`を置かない(撤去) ③全アプリが自前紐付けを持つ状態を維持。手順は `server-app-deploy` スキルに定型化済み。
- **[Vercel]** 環境変数を変えた/コードを編集した後は**必ず再デプロイ**。**編集前のデプロイは反映されない**(古い設定のまま本番に出る)。順序は「編集→コミット→再デプロイ」。
- **[Vercel]** 環境変数の登録: 新UIは Settings→Environments→Production を開く。既存キーは**新規追加でなくEdit**で値を入れ替える(重複追加はエラー)。Sensitiveでも上書き可。**APIキーはチャットに貼らずサイト画面に直接**。
- **[デバッグ]** 本番500は推測で直さず**Vercelランタイムログを先に見る**(MCP get_runtime_logs)。`AuthenticationError`=キーの値が無効(空白混入/別アカウント/失効)、`...KEY missing`=未登録or再デプロイ未実施、と切り分けられる。
- **[Firebase]** 既存アプリのFirebase設定(`firebaseConfig`)はコードに直書き済み=流用可能。`firebaseConfig`は公開設定で秘密ではない。ただし複数アプリの分離を重視するなら専用プロジェクトを作る方が綺麗(あいだは専用 aida-3cc6f を採用)。
- **[Firebase]** Firestoreの書き込み確認は、Web APIキー付きのREST(`firestore.googleapis.com/.../documents/[col]?key=...`)でClaude側から疎通テストできる(ルールが `if true` なら成功)。

## 2026-06-16: ReceiptWarikan / QRLivePoll
- **IME / Enterハンドラ** 入力欄でEnterキーに「項目追加」等を割り当てると、日本語IMEの変換確定Enterを誤検知して未確定の文字で項目が作られる。`if (e.isComposing) return;`（または `e.keyCode === 229` 判定）で変換中を除外する。確定後のEnterだけが発火するようになり、欄も正しくクリアされる。日本語UIの全アプリで踏むので定型対策にする。
- **金額表示の正確性** 各人の金額を独立に `Math.round` すると合計が総額とズレる（例: 1000円を3人→各333円表示で合計999円）。お金を扱うUIでは許容不可。**最大剰余法**で端数を1円ずつ配分し「各人表示額の合計＝総額表示」を保証する（詳細は reusable-patterns）。

## 2026-06-28: RideDex / TrainDex（LLM Vision の限界）
- **LLM Vision / fine-grained同定** OpenAI Vision は「細かい同定」と「座標出力」が構造的に苦手。日本車の世代・鉄道形式のような専門的同定は gpt-4o-mini はもちろん **gpt-4o でも外す**（N700SをN700系と誤答＝Sを落とす）。ナンバープレート/顔の座標も両モデルとも `[]` を返しがちで、**LLM座標ベースの自動マスキングは当てにならない**。→ ①fine-grainedな正解性をLLMに保証させない＝「当てる」より「**AI推定＋人が訂正して育てる**」設計にする（確度<60%は「自信なし」を明示し断定トーンを排除）。②「自動で隠す」より「**手動でなぞって確実に隠す**」を主役にする。モデル格上げ（mini→4o）では精度もコストも割に合わない。
- **検証** 「動いた」と報告する前に**正解の分かる実物データで測る**。雑な図形テストでは精度もぼかしの実効も分からず、Keiの実機報告で初めて露見した。Playwrightの図形通過≠実機で機能する。
- **名寄せ / 重複** 同じ対象でもAIが英語(Toyota/Prius)と日本語(トヨタ/プリウス)で揺れて別キー→「初発見」重複が起きる。対策2層：①プロンプトで maker/model を**日本語固定**（根本を断つ）②キー正規化を NFKC＋空白/記号/中黒/長音/括弧除去で強化。文字種違いは辞書なしでは完全名寄せ不可なので、出力言語をプロンプトで固定して発生源を断つのが実用解。

## 2026-07-10: 引き継ぎ職人（Day36）
- **日付計算 / 期限判定** `Math.ceil` は負のゼロ(-0)を返すことがあり `>= 0` 判定を通過する＝期限が昨日でも「期限は今日！」と表示される。期限系UIは**超過判定を先に**行い、日数差の丸めはその後にする（締切のあるアプリ全部で踏む）。

## 2026-07-14: 手放しトリアージ（Day40）
- **数値入力の「0」とfalsy判定** 0が正当な入力になる欄（売値0円・残数0個など）で `!value` や `Number(value) || 0` だけで必須チェックすると、0入力が「未入力」と誤判定されて先に進めない。**必須判定は raw の `value === ''` で行い、数値化（`Number()`）はその後**に分離する。手放しトリアージでは「売れそうにない物こそ0円で判定したい」がコア動線だったので、このバグはコア価値を直撃していた（QAで検出→修正→0円が「手放す」判定に進むことを実測）。金額・回数・年数を扱う全アプリで最初に確認する。
