## 2026-06-06 夜間リサーチエージェント設計
**決定**: 毎晩0:00 JSTにRemote Routine（CCR）でWebリサーチを自動実行し、`企画部/research/YYYY-MM-DD.md`に企画ヒントつきレポートを生成する
**理由**: 毎日10案を考えるときの情報源を自動化することで、企画の質と速度を上げるため。人が毎晩リサーチするコストを削減し、朝の「おはよう」フローに自然に組み込める
**却下した案**: ローカルcronで実行（Macがスリープ中は動かない）、毎朝手動でリサーチ（毎日のルーティンにするには重い）

## 2026-06-06 GitHub連携方針
**決定**: GitHubリポジトリを「デプロイ起点 + ポートフォリオ」として整備する。Vercelはリポジトリ連携で自動デプロイ、FirebaseはGitHub Actionsで自動デプロイ
**理由**: 手動デプロイは忘れやすく、100日間継続するには自動化が必要。ポートフォリオとして毎日のコミット履歴が残るメリットもある
**却下した案**: 今のまま手動デプロイ継続（1日1プロジェクトのペースでは運用コストが高い）

## 2026-06-05 FoodScore — Claude Vision API構成
**決定**: 1日プロジェクトのMVPとして、Vanilla JSからClaude Vision APIを直接呼ぶ構成で実装した。
**理由**: 料理写真アップロード、画像リサイズ、AI採点、Chart.js表示、PWA、Firebase Hostingまでを1日で通すことを優先したため。
**却下した案**: 初日からFirebase Functions / Cloud Runを挟む構成。APIキー管理としては正しいが、MVPの学習範囲が広がりすぎるため次回改善に回した。

## 2026-06-05 FoodScore — OpenAI + Vercel APIプロキシ移行
**決定**: 友達共有URLでは、ブラウザから直接AI APIを呼ばず、Vercel Serverless Function経由でOpenAI Visionを使う構成に移行した。
**理由**: 最終成果物を「友達にURLだけ渡せる状態」と定義したため。APIキー入力を友達に求めず、キーをVercel環境変数でサーバー側管理する必要があった。
**却下した案**: Firebase Functionsでの実装。無料枠ではBlazeプランが必要だったため、今日の少人数共有にはVercelの方が現実的だった。

## 2026-06-05 Daily Control Sheet
**決定**: 日報とは別に `運用部/daily/YYYY-MM-DD.md` を作り、その日の選定・状態・URL・未完了項目を管理する。
**理由**: セッションログ、日報、QA、広報、記憶庫の状態がズレたときに、1日の正本として参照できる場所が必要だったため。
**却下した案**: 日報だけで管理する方式。日報は読み物・同期トリガーの役割もあるため、状態管理のチェック表としては不十分。

## 2026-06-05 100日チャレンジ — 最終成果物の定義
**決定**: 今後の最終成果物は「友達にURLを渡したら、追加設定なしで主要機能を使える状態」とする。
**理由**: コードが動くことと、人に渡せることは別。100日後に残したいのは制作物の断片ではなく、実際に触れる小さなプロダクト群だから。
**却下した案**: ローカルで動く・デプロイURLだけある・APIキーを友達に入力してもらう状態を完成とみなす方式。

## 2026-06-05 100日チャレンジ — UI/UX改善3ループ
**決定**: デプロイ後、デザイン部がWebトレンド調査を行い、First Impression / Core Flow / Shareability の3ループで改善議論・実装・再デプロイ・確認を行う。
**理由**: 1日1プロジェクトでは「作る」だけに寄りやすい。友達に渡す前提なら、初見の分かりやすさ、主要導線、共有したくなる完成度を短時間で磨く仕組みが必要。
**却下した案**: QAだけでUIを見る方式。QAは壊れていないかを見る役割で、体験の魅力や共有されやすさを磨くには専用のデザイン視点が必要。

## 2026-06-04 日報フォーマット設計
**決定**: 日報はObsidian daily noteに全文直接書き、「今日の学び」はユーザー向けのみ。Claude向けはretrospectiveで記憶庫へ
**理由**: 2ファイル管理の複雑さを排除。学びの対象（人間 vs AI）を分離することで日報が振り返りとして機能するようになる
**却下した案**: 運用部/日報/をObsidianにエンベッド（Vault外参照不可）、学びをまとめて書く（Claude向け技術メモが混在して読みにくい）

## 2026-06-04 MCP認証情報の管理
**決定**: macOS KeychainにPATを保存し、`claude mcp add`で`~/.claude.json`に登録
**理由**: settings.jsonに直書きするとファイルコピー時に漏洩するリスクがある
**却下した案**: settings.jsonのenvフィールドにトークンを直書き（セキュリティリスク）

## 2026-06-04 記憶庫の設計：ファイルベース vs 外部サービス
**決定**: `記憶庫/` ディレクトリ内のmarkdownファイル（decisions.md / lessons.md / reusable-patterns.md）で知見管理
**理由**: Claude Code のコンテキストはファイル読み込みで補完される。Git管理下のmarkdownなら、CLAUDE.md等と同じ仕組みで参照でき、ツール追加ゼロで運用できる。Mem0/Zep等の外部サービスはAPI連携コストがあり、1人スタジオには過剰。
**却下した案**: 外部記憶サービス（Mem0等）→ API連携・認証・コスト管理が必要。セッションメモリのみ → セッション間で消える。

## 2026-06-01 whisky-note — スタック選定：vanilla JS + Firebase
**決定**: ビルドステップなしの vanilla JS + Firebase CDN を採用
**理由**: 1日開発で即デプロイが目標のため、バンドラー設定に時間を使いたくない。`firebase serve` → `firebase deploy` で完結する最短経路を取った。FirebaseのCDN版（compat SDK 10.12.0）は import/export 不要でグローバル参照でき、`<script>` タグ追加だけで機能が増やせる。
**却下した案**: React + Vite → ビルド設定・HMR・npm依存の管理コストが1日スプリントには過剰。TypeScriptも同様の理由で却下。

## 2026-06-01 mood-forecast — スタック選定：React + Vite + localStorage
**決定**: React + Vite で SPA を構築し、データは localStorage のみに保存
**理由**: mood-forecast は認証不要・クラウド同期不要のローカル完結アプリ。Chart.js等のnpm依存を使いたかったため、バンドラー（Vite）が必要だった。Firebaseを入れるとコスト（Auth設定・Firestore rules）が発生するが、ユーザー価値の増分がゼロなので却下。
**却下した案**: vanilla JS → chartsライブラリのESM依存解決が煩雑。Firebase → 認証・クラウド同期が不要なユースケースにオーバースペック。
