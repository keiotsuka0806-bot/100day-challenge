# Genius Audit — 天才アプリ開発者レビュー（デプロイ前ゲート）

**デプロイ前**のアプリを「天才アプリ開発者」の目で解剖し、致命的な穴を見つけて即実装する。
UX改善はデザイン3ループで行うため、ここでは**セキュリティ**と**コアバリュー欠落**のみに集中する。
設計論の議論ではなく、**今すぐ直す**こと。

## 起動タイミング

- 実装完了・**デプロイ前**（ステップ3）← メインの起動タイミング。通過するまでデプロイ不可
- 「天才アプリ開発者として分析して」と言われたとき
- デプロイ済みアプリへの追加分析が必要なとき（デザインループ外の問題発覚）

---

## Step 1: 全ファイルを読む

以下を必ず読む。読まずに診断を始めない。

```
index.html      ← UIの全構造、欠けているセクション
app.js          ← データフロー、API呼び出し、エラー処理
styles.css      ← デザイントークン、モバイル対応
api/*.js        ← サーバーサイド処理、セキュリティ
vercel.json / firebase.json ← デプロイ設定
```

デプロイ先URLを確認する。ない場合は .firebaserc / vercel.json / .vercel/project.json を読む。

---

## Step 2: 天才視点で診断する

### 診断の核心的な問い

> 「友達にURLを渡したとき、このアプリはその人の人生に何かを与えるか？」
> 「アプリのコアバリューを妨げている障害は何か？」
> 「なぜ今これを作るのか。他と何が違うか。1文で言えるか？」

### チェックリスト（全項目を必ず確認）

**🔴 セキュリティ**
- APIキーがクライアントに露出していないか（localStorage / HTML直書き）
- XSS: innerHTML にエスケープなしのユーザー入力が入っていないか
- APIにレート制限・サイズ制限はあるか

**🔴 コアバリューの欠落**
- アプリのメインコンセプトに対して欠けている機能はないか
  - SNSアプリ → シェア機能がない？
  - 記録アプリ → 履歴がない？
  - 採点アプリ → 比較・推移がない？
- エラーが `alert()` で表示されていないか（インラインUIに変える）

**🟡 UX**
- ローディング状態・エラー状態・空状態はすべて網羅されているか
- モバイルでの操作性（タップエリア、フォント、スクロール）
- フィードバックループ：ユーザーは「次に何をすればいいか」分かるか

**🟡 データ**
- 履歴・スコア・入力データの永続化は適切か
- 取得したデータを薄くしか保存していないケース（後から詳細を見たいときに困る）
- Service Workerのキャッシュ戦略は正しいか

**🟢 アーキテクチャ（致命的でなければコメントのみOK）**
- APIキーのサーバーサイド化は済んでいるか
- 外部APIのレスポンスを正規化・バリデーションしているか
- 画像・ファイルのサイズ制限はあるか

## Step 3: 構造化して出力する

以下のフォーマットで必ず出力する。省略禁止。

```
## [プロジェクト名] 天才レビュー

**診断サマリー**
（2〜3文で核心的な問題を述べる。「〜なのに〜がない」という矛盾を指摘する形が最も鋭い）

### 現在のワークフロー
[入力] → [処理A] → [処理B] → [出力]
                                  ↓（欠落）
                              [本来あるべき後処理]

### 理想のワークフロー（実装後）
[入力] → [処理A] → [処理B] → [出力]
  ├── [追加機能1]              ← NEW
  └── [追加機能2]              ← NEW

### 修正点一覧
| 優先度 | 問題 | 影響度 |
|--------|------|--------|
| 🔴 | ... | 致命的 |
| 🟡 | ... | 高/中 |
| 🟢 | ... | 低（将来） |

### 必要スキル・MCP
| 目的 | ツール | 理由 |
|------|--------|------|
| ... | Vercel MCP / Playwright MCP / /code-review | ... |
```

---

## Step 4: 実装する

**🔴 Priority 1（致命的・セキュリティ・コアバリュー欠落）は必ず実装する。**
**🟡 Priority 2（UX・データ）は可能な限り実装する。**
**🟢 Priority 3（アーキテクチャ）はコメントと設計案のみでOK。**

実装時の原則：
- 既存のコーディングスタイル・変数名規則を踏襲する
- ファイルを読んでから編集する（Edit ツールを使う）
- 追加機能はコアフローを壊さない位置に差し込む
- 日本語UI統一

---

## Step 5: デプロイ可否を判断し、フラグを記録する

**全ての🔴 Priority 1が修正済み**であることを確認してから「デプロイOK」を表明する。

```
✅ セキュリティ: APIキー露出・XSS リスクなし
✅ コアバリュー: 仕様書の「コアバリュー必須機能リスト」がすべて実装済み
✅ エラーハンドリング: alert()が残っていない
→ デプロイ可。QA部（/code-review）に渡す。
```

🟡 Priority 2（UX・データ）はデザイン改善ループで対処するため、ここではブロックしない。

**デプロイ可の判断後、必ず以下を実行する（Pre-deployゲートを解除するため）:**

```bash
touch .genius-audit-ok
```

これを実行しないと `firebase deploy` / `vercel --prod` がhookによりブロックされる。

---

## Step 6: 記録する

1. セッションメモ（サイレント実行）:
```bash
echo "[HH:MM] [プロジェクト名] — genius-audit実施: [主な改善3点]" >> "運用部/sessions/$(date '+%Y-%m-%d').md"
```

2. 重要なパターンがあれば `記憶庫/reusable-patterns.md` に追記する。

## 天才開発者の姿勢

**矛盾を見つける**: 「SNSアプリなのにシェアがない」「採点アプリなのにスコア推移がない」——この種の矛盾がいちばん重要な穴。

**実装するか、しないか**: 「将来的には〜」は禁句。Priority 1は今すぐ直す。できないなら「将来課題」として記録してStep 6で終わる。

**友達に渡せるか**: 最終判断基準はこれ。URLを渡して、追加設定なしで主機能が使えるか。
