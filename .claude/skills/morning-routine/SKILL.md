# Morning Routine（朝会）

2026-06-15のAnthropic課金変更（ヘッドレス実行が月次クレジット制になる）に伴い、launchdの夜間自動実行を廃止して対話実行に統合したもの。対話セッションは課金枠の対象外なので、このスキルは追加コストなしで動く。

## 実行原則（コスト最小）

- 4つの業務（記憶庫レトロ・企画10案・組織改善・note下書き）を**この1セッションでまとめて**実行する。業務ごとに別セッションを立てない
- WebSearchはセッション全体で**3回まで**（企画トレンドリサーチに使う）
- やることがないステップは理由を1行報告してスキップする
- commit/pushは全ステップ完了後に**スコープ単位で**行う（下記Step 5）
- 企画10案は**1案ずつ確認を取らず一気に生成**し、結果は最後の完了報告で**表にまとめて**提示する（Step 6）

## Step 1: 記憶庫レトロスペクティブ（昨日分）

1. `git log --oneline --since='yesterday 00:00' --until='today 00:00'` で昨日のコミットを確認
2. `運用部/sessions/`（昨日の日付）と `QA部/review-*.md`（昨日の日付）があれば読む
3. 昨日のコミットが0件かつセッションログも空ならこのステップは終了

学びがあれば3分類で追記する。追記フォーマットは `.claude/skills/retrospective/SKILL.md` の Step 3 に従う：

- `記憶庫/lessons.md` — 技術的ハマりポイント
- `記憶庫/reusable-patterns.md` — 再利用できるパターン
- `記憶庫/decisions.md` — 意思決定の記録

## Step 2: 企画部 — 今日の企画10案

1. **トレンドリサーチ**（WebSearch最大3回）: Qiita「作ってみた」人気記事 / Zenn個人開発トレンド / GitHub trending・Product Huntのいずれか
2. **重複回避**: `企画部/specs/` の直近14日の企画テーマ**＋本社CLAUDE.mdの既存プロジェクト一覧（デプロイ済み含む）**を確認する。テーマだけでなく「同じ体験の形か」で見る（例: あいだ＝二人の相互理解／SoundFrame・melo-doodle＝音楽制作）。**過去案・既存アプリと同じ形のものは出さない**（2026-06-17、再提案でうっかり`あいだ`と重複した実例より）
3. **生成**: 今日の日付で `企画部/specs/YYYY-MM-DD-[meaningful-name].md` を10件作成（記憶庫decisions 2026-06-09の決定により、idea1〜10の連番は使わない。ファイル名は内容がわかる英語名にする）

各ファイルのフォーマット：

```markdown
# [企画名]（[英語名]）

## 概要
[1〜2文]

## MVP機能
- 機能1〜3

## 技術スタック
- フロントエンド / データ / 特殊ライブラリ

## 難度
**[低（1〜2h）/ 中（半日）/ 高（1日フル）]**

## 独自性
「[類似サービス]に似ているが、[この点が違う]」を1行で

## 20年後の視点
2045年の世界でこの企画はどう見えるか（1文）

## 備考
```

選定基準: 難度は**低〜高を混ぜる（低の数を固定しない。「低5案必須」は廃止）**／トレンドを2〜3案に反映／直近14日・既存アプリと重複しない／2〜3案はClaude APIを使う企画／コモディティ化したジャンル（ToDo・タイマー等）は独自の切り口がない限り不採用。

**最重要・心を動かす軸（2026-06-17 Day 17の教訓）**: 「便利か」だけで選ばない。本社CLAUDE.md絶対ルール3に従い、**見知らぬ誰かが心を動かされ、スクショして、誰が作ったかを覚えるか**を最上位の評価軸にする。
- 10案のうち**少なくとも半分は、Kei自身の作業を楽にする道具ではなく、他人の心を動かす／シェアされる体験**にする（OGP生成・アイコン生成・整形ツール等の「店主の道具箱」ばかりにしない）
- **形の多様性**: 「テキストを貼る→AIや整形が返す」形ばかりにしない。身体・センサー（声・カメラ・傾き）／他者とのつながり／時間・意味・休息など、**入力と体験の形を必ず散らす**
- 着手前に `kei-persona.md` の鉱脈（音楽・水と光／メンタルヘルス・セルフコンパッション／他者を理解すること／働くことの意味）を1度見て、最低2〜3案はどれかに根ざす

**乗り換え理由ゲート（2026-06-12 Day 13の教訓・2026-06-17改訂）**: 各案の「独自性」は機能の差分ではなく**使う/触れる理由**で書く。次の**どちらか**を1文で書けない案は不採用——(a)Keiが既存の定番ではなくこれを使う具体的な場面、**または** (b)見知らぬ誰かがこれに触れて心を動かされ人に見せたくなる理由。※(a)だけを強制すると発想が「店主が今週使う道具」に潰れる（2026-06-17の退屈な10案の主因）。同日に2本（ShortsStudio=CapCutとの違い不明 / AsaBrief=日報を読み直すのと違い不明）が即日ドロップされた実例も参照。

4. **絞り込み（2026-06-15からの新ルール）**: 10案を出したら終わりにせず、企画部自身が**比較して最後の3案まで絞り込む**。
   - 出した10案を横並びで比較し、「自分（企画部）なら絶対に選ばない案」を**理由付きで**1つずつ落としていく
   - 落とす理由は乗り換え理由ゲート・コモディティ度・実装難度に対する価値・独自性の弱さなどを根拠に、毎回**必ず1〜2文で明記する**（「なんとなく弱い」は不可。何がダメで誰の何の場面に刺さらないのかを書く）
   - 10案 → 7案 → 5案 → … と段階的に落とし、**最後の3案**が残るまで続ける
   - 残った3案には「なぜ残したか（誰のどの場面に効くか）」も1行添える
   - 落とした7案・残した3案・各理由は完了報告（Step 6）に**全部書く**。spec mdファイルは10件すべて残す（落とした案も記録として保持し、ファイル削除はしない）

5. **企画10案を日報ファイルに記録（必須・絶対に飛ばさない）**: 10案を Obsidian日報 `/Users/kei/Documents/Kei/日報/YYYY-MM-DD.md` に **Markdown表** で必ず書き込む（無ければ日報テンプレから作成し、`## 企画10案（朝会）` セクションを追加）。これは Step 6 の**チャット完了報告とは別物**で、こちらは「ファイルへの永続記録」。**この工程が無くて 2026-06-16/17 の日報から表が丸ごと欠落した**ので、必ず実行する。
   - 列は固定: `| # | 企画名 | 難度 | 独自性（使う理由）1行 | 判定 |`（罫線ASCIIアートは使わずMarkdownの`|`区切り表。詳細は [[idea-10-recording-rule]]）
   - 朝の時点では判定列に**★最終3案 / 落選**を入れる。その日のアプリを作り終えたら（または「日報更新」時に）判定を**史実**に更新する＝作った案=`★採用(制作)`・作ってドロップ=`制作→Drop`・未制作=`落選`。推測で最終結果を先に書かない／作った後に最終3案のまま放置しない
   - 10案で納得できず別案を作った日は、落選した元10案＋実際に作った案の両方を載せる（行数は10超可）
   - 掲載先は**Obsidian日報のみ**。`運用部/daily/`（作業管理シート）にはこの表を入れない

## Step 3: 組織改善レポート

外部コンサルタント視点で分析し、`運用部/reports/org-improvement-YYYY-MM-DD.md` に保存する。

1. 現状把握: `git log --oneline -20` / `運用部/sessions/` 最新 / `QA部/` 直近レビュー / CLAUDE.mdのプロジェクト一覧
2. 分析: スピード・リズム / 品質トレンド / 技術的観察 / 組織の健全性
3. 明日のアクションアイテム1〜3個

フォーマット: `# 組織改善レポート YYYY-MM-DD` → `## エグゼクティブサマリー` → `## 分析結果`（4小節）→ `## 明日のアクションアイテム`

> **Job Scoutは朝会から分離済み**。求人スカウトは毎朝ではなく、Keiが「Job Scoutして」と頼んだときに `job-scout` スキルで実行する。

## Step 4: 広報部 — note下書き（マネタイズ軸）

「非エンジニアがAIと会社を経営する」を読者に届けるnote記事の下書きを**毎朝1本**生成する。広報部の役割定義は `広報部/CLAUDE.md` に従う。

1. **素材集め**: 昨日の `運用部/daily/`・記憶庫の新エントリ・`git log` から「昨日いちばん物語になる出来事」を1つ選ぶ（失敗談・意外な発見・組織が変わった瞬間が最良。成功自慢より learning が刺さる）
2. **重複回避**: `広報部/drafts/` の既存下書きとテーマが被らないこと
3. **生成**: `広報部/drafts/YYYY-MM-DD-note-[英語slug].md` に保存

下書きのフォーマット：

```markdown
# タイトル案（3つ。数字や具体を入れる）

## リード（無料部分の冒頭。3〜4文で「読む理由」を作る）

## 本文構成
（見出し＋各見出しの要点を箇条書き。本文は7割まで書いてよい）

[Keiの言葉: ここに当時の気持ち・実体験を1〜3文]
（↑Keiにしか書けない部分は必ずプレースホルダで空けておく。勝手に感情を創作しない）

## 想定読者と狙い
（N=1ペルソナ1行 / 無料 or 有料(価格)の提案 / 理由）
```

注意: これは**完成記事ではなく下書き**。Keiが自分の言葉を足して公開するまでが広報部の仕事の半分、という前提で作る。

## Step 5: commit & push

メインworktreeで直接作業しているため、**自分が触ったファイルだけ**をスコープ単位でcommitする（`git add -A` は使わない。Keiの作業中ファイルを巻き込まないため）：

```bash
git commit -m "朝会: 記憶庫レトロ YYYY-MM-DD" -- 記憶庫/
git commit -m "朝会: 企画10案 YYYY-MM-DD" -- 企画部/
git commit -m "朝会: 組織改善レポート YYYY-MM-DD" -- 運用部/reports/org-improvement-YYYY-MM-DD.md
git commit -m "朝会: note下書き YYYY-MM-DD" -- 広報部/drafts/
git push
```

変更がないスコープはcommitしない。pushが拒否されたら `git pull --rebase` してから再push。

## Step 6: 完了報告

冒頭に企画10案を**表でまとめて**提示し、続けて**絞り込みの過程（落とした理由・残した3案）**を必ず見せる（1案ずつの確認は不要。生成済みの結果だけを見せる）：

```
🌅 朝会完了（YYYY-MM-DD）

### 企画10案
| # | 企画名 | 難度 | 独自性(使う理由)1行 | 判定 |
|---|--------|------|---------------------|------|
| 1 | ... | 低 | ... | ★最終3案 / 落選 |
| … | … | … | … | … |
| 10 | ... | 高 | ... | ... |

### 絞り込み（10案 → 3案）
落とした7案（理由付き）:
- [企画名]: [絶対に選ばない理由 1〜2文]
- …（7件）

最終3案（残した理由付き）:
1. [企画名] — [誰のどの場面に効くか 1行]
2. ...
3. ...

- 記憶庫: [追記件数 or スキップ理由]
- 組織改善: [サマリー1行]
- note下書き: [タイトル案1つ目]（広報部/drafts/...）
特に重要: [今日いちばん価値ある知見を1行]
```

note下書きは「あとはKeiの言葉を足すだけ」の状態でパスを案内すること。
