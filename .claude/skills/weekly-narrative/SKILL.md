# Weekly Narrative（週次ナラティブ）

毎日の `morning-routine` Step 4 が「昨日1日分のnote下書き」を作るのに対し、このスキルは**1週間分を蓄積素材として束ね、起伏のある1本の物語**に編み直す。週次でやることで情報密度が上がり、単発の日次下書きでは出せない「葛藤→転機」のアークが立つ。

WeekNarrative企画（`企画部/specs/2026-06-15-week-narrative.md`）は、公開アプリではなく**Keiが自分で使う仕組み**として実装すべき、という設計判断（2026-06-15）からスキル化したもの。対話セッション内で動くので追加コストはかからない。

## 実行原則

- 対象は**直近7日**（呼び出し日の前日までの1週間）。引数で期間を変えてもよい
- WebSearchは使わない（社内の記録だけで完結する）
- 出力は `広報部/drafts/YYYY-MM-DD-weekly-[英語slug].md` に保存する
- これは**下書き**。Keiが自分の言葉を足して公開するまでが半分（広報部ルールに従う）

## Step 1: 素材集め（直近7日）

1. `git log --oneline --since='7 days ago'` でその週のコミットを取得し、`autosave:` 等のノイズコミットは除外する
2. `運用部/daily/`（直近7日分の日報）を読む
3. `運用部/sessions/`（直近7日分）と記憶庫の新エントリがあれば拾う
4. その週の「事件・つまずき・意外な発見・組織が変わった瞬間」を時系列でメモする（成功自慢より learning が物語になる）

## Step 2: 物語アークに再構成

集めた素材を、単なる要約ではなく**起伏のある物語**に編む。4段で構成する：

- **発端** — その週、何から始まったか
- **葛藤** — どこでつまずいた・迷ったか（ここが記事の心臓。1週間ぶんの中で最も痛かった/驚いた瞬間を選ぶ）
- **転機** — 何がきっかけで変わったか
- **学び** — 結局、何を持ち帰ったか（Keiの worldview に接続できるとなお良い）

注意：嘘をつかない（絶対ルール）。素材に無い感情やドラマを創作しない。Keiの内面は `[Keiの言葉: ...]` プレースホルダで空ける。

## Step 3: 記事骨子を出力

`広報部/drafts/YYYY-MM-DD-weekly-[英語slug].md` に保存する：

```markdown
# タイトル案（3つ。数字や具体を入れる）

## リード（無料部分の冒頭。3〜4文で「読む理由」を作る）

## 本文構成（発端→葛藤→転機→学び）
（各段の見出し＋要点を箇条書き。本文は7割まで書いてよい）

[Keiの言葉: 当時の気持ち・実体験を入れる場所を1〜3箇所、プレースホルダで空ける]

## 今週の事実ログ（裏付け）
（記事の根拠にした実際の出来事・コミットを箇条書き。捏造防止のトレーサビリティ）

## 想定読者と狙い
（N=1ペルソナ1行 / 無料 or 有料(価格) / 理由。マネタイズ軸=note発信→メンバーシップ→プロダクト）
```

## Step 4: commit

```bash
git commit -m "週次ナラティブ: YYYY-MM-DD週" -- 広報部/drafts/
```

## Step 5: 完了報告

- タイトル案3つを提示
- 「今週いちばんの物語の山場（葛藤→転機）」を1行で
- Keiが言葉を足す箇所（プレースホルダの数と位置）を案内
- 保存パスを案内
