# Daily Autopilot（自走ループ・司令塔）

一日の工程を1本のセッションで順に流す。**自分で実装ロジックを持たず、既存Skillを呼ぶだけ**（重複させない）。各Phaseの後で**必ず停止して承認を取る**のが本スキルの本質。

## 大原則（コスト・安全・自走の定義）

- 全工程を**この1対話セッション内**で回す＝**追加課金なし**。headless・cron・GitHub Actions・Agent SDKは作らない・使わない（[[billing-audit-2026-06]]）。
- **自動でやらないこと**: 本番デプロイ、外部投稿(note/X)、秘密情報の書き出し、commit/push。これらは**必ず人の承認・操作を挟む**。
- 「自走」とは**無人**の意味ではなく、**人が各判断だけすれば工程が途切れず流れる**状態を指す。
- WebSearch予算はセッション全体で**3回まで**（morning-routine と共有）。
- 全Phaseを毎回やる必要はない。やることが無いPhaseは理由を1行報告してスキップ。途中Phaseから開始・再開してよい。

## Phase 0: 状態確認（着手前）

1. `運用部/project-registry.json` と直近の `運用部/daily/*.md` を読み、昨日の残タスク・進行中プロジェクトを把握。
2. 今日 `date +%F` を確定。今日どこまでやるか（全工程か、制作だけか等）を**Keiに1行確認**してから進む。

## Phase 1: 朝会（企画）

- **`morning-routine` スキルを呼ぶ**（記憶庫レトロ・企画10案・組織改善・note下書きを実行）。
- 企画10案が出たら **▲承認P1**: 「今日はどの案を作る？」と聞き、**Keiが1案選ぶ**まで先に進まない。

## Phase 2: 制作

- 採用案について **`project-kickoff` スキルを呼ぶ**（プロジェクト雛形・配置）。
- 実装はKeiとの協働で進める。ローカルプレビューが動く状態をゴールにする。
- 既知の罠を先回り: Vercel SSO / モノレポ誤爆 / Service Workerキャッシュ / スマホ320px崩れ（各メモリ参照）。

## Phase 3: 3ゲート（デプロイ前・必須）

`運用部/CLAUDE.md` のゲート順に**この順で**実施。1つでも未通過ならPhase 4に進まない（[[predeploy-gate-rule]]）:
1. **Genius Council / genius-audit（ゲート1）** — ローカルプレビューに対して3ループ改善。
2. **QA / `/code-review`（ゲート2）** — 正確性バグを確認。記録を `QA部/review-YYYY-MM-DD.md` に。
3. **セキュリティ / `security-check` スキル（ゲート3）** — 鍵露出/XSS/インジェクション等。通過後はコードを変えない。
- **▲承認P2**: 3ゲートの結果を要約提示し、Keiの「デプロイGO」を待つ。

## Phase 4: デプロイ（人の操作が必須）

- サーバー機能つき（AIキーを隠す）アプリは **`server-app-deploy` スキルを呼ぶ**。静的アプリはFirebase Hosting。
- **ログイン・APIキー登録・Vercelコンソール操作は人しかできない**。Claudeはファイル操作・curl確認・ログ確認まで。
- 公開後: `curl -sI <url>` で **HTTP 200** と **SSO壁が無いこと** を実測（[[vercel-sso-protection-gotcha]]）。
- **▲承認P3**: 公開URL・実測結果を提示し、Keiの「公開OK」を確認。

## Phase 5: 記録（日報）

- **`daily-report` スキルを呼ぶ**（repo `運用部/daily/` と Obsidian `日報/` を両方更新。既存非空ファイルは**追記**）。

## Phase 6: 発信下書き（投稿はしない）

- noteは Phase 1 の morning-routine で下書き済みのはず。無ければここで `広報部/drafts/` に1本生成。
- **`x-draft` スキルを呼ぶ**（当日成果＋note下書きから X投稿案を生成・保存）。
- **▲承認P4**: 「下書きを用意しました。投稿はKeiが手で」。**このスキルは投稿ボタンを押さない**。

## Phase 7: 締め

1. 学びがあれば **`retrospective` スキル**で記憶庫3ファイルに追記。
2. `運用部/project-registry.json` を当日状態に更新。
3. **▲承認（commit）**: 変更スコープを提示し、Keiの確認を得てから `commit/push`（スコープ単位。日報・広報・開発部は混ぜない）。

## 完了報告

- 実行したPhaseと各承認ポイントの結果を、**表または3〜5行**で要約する。
- スキップしたPhaseは理由を1行。未通過ゲートや未対応TODO（例: OpenAI月額上限設定）は残タスクに明記。

## やらないこと（ガードレール総括）

- 無人自動実行・cron・課金を伴う実行は**作らない**（要る時はKeiに別途コスト確認）。
- 本番デプロイ・note/X投稿・commit/pushは**人の承認なしに実行しない**。
- 秘密情報（`~/.secrets/keys.env` 等）をチャットや成果物に書き出さない。
