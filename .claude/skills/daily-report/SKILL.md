# Daily Report（一日の締め・二重更新の自動化）

日報は2箇所に書くのが正本（メモリ [[daily-report-dual-update]]）:
- repo: `運用部/daily/YYYY-MM-DD.md`（詳細版 = Daily Control Sheet）
- Obsidian: `/Users/kei/Documents/Kei/日報/YYYY-MM-DD.md`（journal = リンク＋1行サマリ）

このスキルは**両方を1セッションでまとめて書く**。片方だけ更新して忘れる事故を消すのが目的。

> **正本は `運用部/CLAUDE.md`「日次レポート形式」**。このスキルはその仕様を対話で確実に実行するための手順書であり、仕様が食い違ったら運用部CLAUDE.mdを優先する。正本では日報更新時に次も一緒に行う: Obsidianプロジェクトノート(Day N)・企画10案テーブル・`project-registry.json`（下記 Step 3b）。Obsidian日報は Vault直下 `日報/YYYY-MM-DD.md` の**1ファイルだけ**（旧・三重書きは廃止）。フォーマットは Vault `テンプレート/日報テンプレート.md` に従う。

## コストと安全（最優先）

- 対話セッション内で完結＝**追加課金なし**。headless実行・cron・GitHub Actionsは作らない。
- 書き込むのは**markdown 2枚だけ**。デプロイ・外部投稿・秘密情報の書き出しは一切しない。

## 既存ファイルの扱い（重要・上書き禁止／新規は追記・重複は更新）

**既存の非空ファイルを丸ごと上書きしてはいけない。**
理由: その日の日報には(1) **日付を跨いで入力された情報**、(2) **Keiが個人で手入力したメモ**が含まれている可能性が高く、機械が消してはいけないため。

書き方は、新しい情報か・既にある情報かで分ける:

- ファイルが**無い** → 新規作成してよい。
- ファイルが**ある & 内容が新規**（既存に無い話題） → **既存の行を1行も削除・書き換えせず**、ファイル末尾に**追記**する:
  ```
  ---
  ## 追記（HH:MM 自動生成）
  [新しい記録]
  ```
- ファイルが**ある & 既存と似た情報**（同じ話題・重複する記述） → 追記で重複させず、その該当箇所を**更新（統合）**する。ただし:
  - 更新は既存行に触れる操作なので、**変更する箇所の差分（before→after）を先に提示し、Keiの確認を得てから**書く。
  - Keiが手入力したメモ・より詳しい記述は**残す**（機械の要約で人の言葉を潰さない）。数値やステータスなど**明らかに更新すべき事実だけ**を新しくする。
- **迷ったら追記に寄せる**（消す/書き換えるより、残す方が常に安全）。
- 非空ファイルに触れる前に「末尾に追記します」または「◯◯の箇所を更新します（差分↓）」と**1行告知**してから書く。

## Step 0: 日付を確定する（暦日基準・案B）

- メモリ [[daily-report-date-convention]] に従い、**その作業が属する暦日**を使う（デプロイがAM跨ぎでも作業本体の暦日で記録）。
- `date +%F` で今日を取得。深夜（0〜4時台）に「昨日の続き」を締めている場合は、**どちらの暦日に付けるかをKeiに1行確認**する。
- 既存の日付ずれ（過去分が1日ずれている未対応TODO）を見つけても勝手に直さず、残タスクに1行メモするだけにする。

## Step 1: 当日の実作業を集める（証拠ベース・推測しない）

実際にやったことだけを材料にする（無い項目は「なし」と書き、創作しない）:

```bash
cd /Users/kei/dev/100day-challenge
git log --oneline --since='today 00:00'          # 当日のコミット
```
- `QA部/review-YYYY-MM-DD.md` … あればゲート2(QA)の結果
- `デザイン部/reports/`（当日）… Genius Council(ゲート1)の記録
- `運用部/project-registry.json` … 当日公開プロジェクトの状態・URL
- 当日の `開発部/[project]/` … 実装内容・API・スタック
- 公開URLがあれば任意で `curl -sI <url> | head -1` で HTTP 200 を実測（read-onlyのみ）

## Step 2: repo側 Daily Control Sheet を書く

`運用部/daily/YYYY-MM-DD.md` に、既存日報と同じ構成で書く（直近の `運用部/daily/*.md` を1つ読んで見出し粒度を合わせる）。**既存ファイルなら上記ルールで末尾追記**:

```markdown
# Daily Control Sheet YYYY-MM-DD

## 今日の主眼
**[何を作り・どこまで通し・公開したか1〜2文]**
> 配置: `開発部/[app]/` 公開URL [url]

## 設計判断（なぜこの形か）
- [ユーザー体験上の判断とその理由]

## 実装（スタック）
- [主要ファイルと役割 / APIの要点]

## デプロイ前ゲート（3つ通過）
- **genius-audit（ゲート1）**: [🔴の有無と対応]
- **QA/code-review（ゲート2）**: [バグ件数・記録先]
- **セキュリティ（ゲート3）**: [点数・検出事項]

## デプロイ記録
| 時刻 | プロジェクト | 内容 | URL | ステータス |
|---|---|---|---|---|

## 課金メモ
- [OpenAI/Anthropic/Vercel/Firebaseの当日コストと未対応の上限設定TODO]

## 学び
- [再利用できる学びを1〜2個]

## 残タスク（次回）
- [ ]
```
- 未実施のゲートは「未通過」と正直に書く。

## Step 3: Obsidian側 日報を書く（mirror）

`/Users/kei/Documents/Kei/日報/YYYY-MM-DD.md` を、**既存テンプレの見出しに沿って**書く。**既存ファイルなら上記ルールで末尾追記**（手入力メモを壊さない）:

```markdown
# YYYY-MM-DD 日報

## 今日の一言
[要点を一人称「私」で1文（[[kei-note-account]] の物語調に寄せる）]

## 今日やったこと
- [3〜5行。repo日報の要約]

## 100日チャレンジ
- [[Day N プロジェクト名]] — [1行サマリ＋公開URL]

## 学んだこと
- [repo日報の「学び」から1〜2行]

## 明日やること
- [ ] [repo日報の残タスクから]

## 数字・記録
- [公開URL / ゲート点数 / コスト等]
```
- Vault直下に `CLAUDE.md` があれば先に従う。
- `Day N` の N は project-registry か直近日報の連番から確定（不明なら1行確認）。

## Step 3b: プロジェクトノート・企画10案・registry（正本の必須項目）

当日に変化（デプロイ・状態変更）があれば、`運用部/CLAUDE.md`「日次レポート形式」に従い次も更新する（**すべて既存を壊さず追記/更新**）:

1. **Obsidianプロジェクトノート**: `/Users/kei/Documents/Kei/100 Day Challenge/プロジェクト/Day N [Project].md`
   - デプロイ済みは `Day N [ProjectName].md`（N は通し番号）、構想段階は `企画 [ProjectName].md`。
   - タイトル直下のサマリー表（本日の選定/完成物/URL/状態/主な技術）は**必須**。状態やURLが変わったら表も更新。
2. **企画10案テーブル**: Obsidian日報に `## 企画10案（朝会）` 表があるか確認。無ければ `企画部/specs/YYYY-MM-DD-*.md` から復元し、判定列を**史実**に更新（作った案=`★採用(制作)`／作ってドロップ=`制作→Drop`／未制作=`落選`）。罫線ASCIIではなくMarkdown表（[[idea-10-recording-rule]]）。
3. **project-registry**: `運用部/project-registry.json` の status / url / lastDeploy / lastReleaseCheck 等を当日状態に合わせる（`node 運用部/scripts/sync-project-registry.mjs` で同期可）。
4. 表の整形（パディング）を**勝手に変えない**。既存ファイルに追記するときは既存の整形に合わせる。

## Step 4: 整合チェック（ここで止めない工夫）

1. **2ファイル両方**が存在するか（片方漏れが最大の事故）
2. 日付が暦日で一致しているか（repoとObsidianで同じ）
3. Obsidianのリンクが `[[Day N プロジェクト名]]` 形式か
4. repo日報のゲート記述が実際の記録ファイルと矛盾しないか
5. 材料が足りない項目を創作していないか（「なし」で埋めたか）
6. 当日変化があったなら Step 3b（プロジェクトノート・企画10案表・registry）も反映したか

不足があれば、勝手に埋めず**何が足りないかを1行で聞く**。

## Step 5: 完了報告（commitは確認してから）

- 「repo日報・Obsidian日報の2枚を更新（新規/追記）。要点は〜」と1〜3行で要約。
- commit/pushは**Keiに一言確認してから**。日報のみのcommitはスコープを `運用部/daily/` に限定（Obsidian Vaultはgit管理外の可能性が高く、保存だけで十分）。

## やらないこと（ガードレール）

- 本番デプロイ・外部投稿・note公開・秘密情報の記載は**範囲外**。
- **既存の非空ファイルを丸ごと上書きしない**（新規情報は追記／似た情報は差分確認のうえ更新）。
